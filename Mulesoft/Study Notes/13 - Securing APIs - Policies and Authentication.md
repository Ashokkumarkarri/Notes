# Chapter 13: Securing APIs — Policies and Authentication

> Taught on: 23-02-2022, 24-02-2022, 26-02-2022, 28-02-2022, 03-03-2022, 04-03-2022, 07-03-2022, 08-03-2022, 10-03-2022, 11-03-2022

## Why This Matters

Chapter 12 covered how RAML lets you restrict *what shape* of data an API will accept — the right method, the right media type. But shape restrictions alone don't answer a much more important question: **who is allowed to call this API at all?** A perfectly well-formed JSON POST request from someone who has no business calling your endpoint is still a problem. That's what this chapter is about — moving from "restrictions" to actual **security**, and understanding the mechanics MuleSoft gives you to enforce it: policies, credentials, and the plumbing (Auto Discovery, Access Management, Exchange) that connects your deployed application to the platform's security rules.

This distinction — restrictions vs. security — was stated explicitly in the course: an API built purely with RAML/API Kit gives you restrictions (method, media type) but not security. Security is a separate configuration layer applied in **API Manager**.

## API Auto Discovery: Linking a Running App to Its Managed API

Before any policy can actually take effect on a *running* application, the application needs to be linked to the API definition that lives in API Manager. That link is called **API Auto Discovery**.

**Auto Discovery** is a component you search for in Anypoint Studio's palette. It connects an API that's designed and managed in **API Manager** to the actual running Mule app deployed in **Runtime Manager**. To configure it:

1. Copy the **API ID** from API Manager (every managed API instance has one).
2. In Anypoint Studio, go to **Global Elements** → create a new **Auto Discovery** component.
3. Enter the API ID, and select the main flow it should apply to.

Once Auto Discovery is wired up, any policy applied in API Manager (Basic Authentication, Client ID Enforcement, etc.) actually gets enforced against the deployed runtime application.

There's an important rule to internalize here: **policies only take effect once the app is deployed via Runtime Manager (CloudHub).** They do **not** apply when you're running or testing the application locally in Anypoint Studio. This has a practical consequence during development — if you want to test your flow logic locally without a policy getting in the way, temporarily remove the Auto Discovery component, do your local testing, then add it back and redeploy so the policy takes effect again in the cloud.

## Basic Authentication Policy

**Basic Authentication** is the simplest of the two policies covered in this course. It's applied in API Manager on top of an API that was designed in Design Center, and it secures the application with a plain **username and password**.

- Without valid credentials, a request returns **401 Unauthorized**.
- To call a Basic-Auth-secured API from Postman: go to the **Authorization** tab, set the type to **Basic Auth**, and enter the username and password directly.

The credentials themselves are entered **directly inside the policy configuration screen in API Manager** — there's no separate credential-generation step. This is one of the two defining characteristics that distinguishes Basic Auth from Client ID Enforcement (the other being explained below).

## Client ID Enforcement Policy

**Client ID Enforcement** is the second policy option, and it works quite differently from Basic Authentication — understanding that difference is the core of this chapter.

With Basic Authentication, you manually create and type in a username/password when configuring the policy. With Client ID Enforcement, the app is instead secured using a **Client ID** and **Client Secret** — but these are **not typed in manually anywhere**. They must be **generated in Anypoint Exchange**, tied to a specific registered client application.

Why does this distinction matter in practice, beyond "different steps"? Client ID Enforcement is considered the more secure of the two options, for a concrete reason: Basic Auth credentials are something *you* create yourself — simple, and potentially left unchanged for a long time (weeks, months). Client ID Enforcement credentials, by contrast, are auto-generated by the backend system, are much longer random strings, and — critically — **cannot be changed once generated; they can only be created once.** That combination (longer, random, non-reusable-by-design) is what makes it the stronger option.

Rule reinforced multiple times across the course: **only one policy can be applied to a given API at a time** — you never run Basic Auth and Client ID Enforcement together on the same API. Which one an organization chooses depends entirely on that organization's own security standard; there's no universal rule dictating one over the other.

### Two Ways to Pass Client ID Enforcement Credentials

The Client ID Enforcement policy supports two different ways a caller can pass its credentials:

1. **Custom Expression** (the default) — client ID and client secret are passed as **separate headers**. The default expression referenced in the course was `attributes.headers.clientId`.
2. **HTTP Basic Authentication header** — the client ID is passed as the *username* and the client secret as the *password*, sent through the standard Authorization header (functionally similar to how Basic Auth credentials are sent, even though the underlying policy is Client ID Enforcement, not Basic Auth).

This matters when testing in Postman: if the Custom Expression option is configured, you pass `client_id` and `client_secret` as separate entries under the **Headers** tab — putting them in the Authorization tab instead will throw an "unauthorized" error. If the HTTP Basic Auth option is configured instead, you use Postman's **Basic Auth** tab, with client ID as username and client secret as password. Other REST clients (SoapUI, Advanced REST Client / ARC) work the same way structurally — same headers, same setup — regardless of which tool you use.

## Configuring and Applying Client ID Enforcement, Step by Step

1. In **API Manager**: **Manage API → Manage API from Exchange**, and search for the API by name (e.g., "Amazon").
2. If the API was only designed in Design Center but never published, publish it first: **Design Center → open the API → Publish to Exchange.**
3. Back in API Manager: **Import from Exchange → Save.**
4. Go to **Policies → Add Policy → Client ID Enforcement.**
5. Choose either the Custom Expression option or the HTTP Basic Authentication header option, then click **Apply.**
6. The policy can be applied either before or after the application is deployed — order doesn't matter.

An important nuance: applying the Client ID Enforcement policy in API Manager only turns the policy **on** — it does **not**, by itself, generate a client ID/secret. Those credentials must be generated separately, in Exchange.

## Generating Client ID and Client Secret in Exchange

1. Go to **Exchange** → open the published API (e.g., "Amazon") → click **Request Access**.
2. Specify the API ID (this is visible on the API's page in API Manager).
3. Check the **Application** dropdown for existing access; if no client app exists yet, create one via **Create New Client App**.
4. Once created, open the application to view its auto-generated **Client ID** and **Client Secret** — copy these for use in Postman or by the actual source system.

If a client ID/secret is ever lost after the client app has already been created, it can be retrieved again: **Exchange → My Applications → select the application → view its credentials.** They can't be regenerated, but they can be looked up again.

## Policy-Level Credentials vs. Access Management-Level Credentials

This is a subtlety that tripped up several students during the course and is worth being very deliberate about, because there are genuinely **two different kinds of "client ID / client secret"** involved in a secured, deployed API, and they serve completely different purposes.

**1. API-level (Client ID Enforcement) policy credentials.** These are the credentials generated in Exchange, described above. The *external caller* (the source system, or Postman standing in for it) must pass this client ID/secret every time it triggers a request into the Mule app. This is what actually enforces "who is allowed to call this API."

**2. Access Management-level (platform) credentials.** These are a completely separate pair, used to link a *Runtime Manager deployment* with *API Manager* — this is what makes Auto Discovery actually work and lets the API show as "Active" rather than "Unregistered." These are declared as **application properties** on the deployed app itself:
- `anypoint.platform.client_id`
- `anypoint.platform.client_secret`

You retrieve these from **Access Management → Environments → select the environment (e.g., Sandbox) → copy the client ID and secret.** In Runtime Manager, you then open the deployed app → **Settings → Properties**, and paste them in under those exact property keys. A small spelling mistake in either property key will prevent the application from deploying correctly — this was flagged explicitly as a real failure mode students hit.

Client IDs at this Access Management level are always different between environments (Sandbox vs. another environment will have different values); platform-level IDs are unique/global to the organization as a whole.

Until these platform properties are correctly configured, the API sits in an **"Unregistered"** state in API Manager. Once configured correctly and the app redeploys successfully, it moves to **"Active"** status, and only then does the policy actually get enforced on live traffic.

To summarize the distinction in one line: **policy credentials (Client ID Enforcement) are what the *caller* must send you; Access Management credentials are what *your app* must present to the platform** so the platform will recognize it and enforce policies on it in the first place.

## Deploying and Testing a Secured Application

Putting the whole chain together, here's the deployment sequence used repeatedly across the course:

1. In Anypoint Studio: right-click the project → **Export** → confirm the local save location → **Finish**. This produces a deployable `.jar` file.
2. Go to Anypoint Platform → **Runtime Manager** → open the target environment (e.g., **Sandbox**).
3. Click **Deploy Application**, give it a name, upload the exported jar, and deploy.
4. The application passes through a **Deploying** state before reaching **Started**.
5. If a platform policy is applied, add the `anypoint.platform.client_id` / `anypoint.platform.client_secret` properties under Settings → Properties (see above) — without them, an app secured by a platform policy will fail to deploy correctly the first time.
6. Once the app shows **Started** in Runtime Manager and the API shows **Active** in API Manager, the policy is actually linked to the live traffic.

Testing from Postman follows the pattern used throughout the course: copy the CloudHub application endpoint from Runtime Manager, replace `localhost`/local port in your request with it, then:

- Sending a request **without** credentials → **401 Unauthorized** — confirms the policy is genuinely active.
- Sending a request **with** the correct credentials (username/password for Basic Auth, or client ID/secret for Client ID Enforcement, in whichever format the policy expects) → request succeeds.

It's worth noting that a policy can be applied to an application either **before or after** it's already deployed — there's no required ordering. However, if you add or change something like Auto Discovery in Studio *after* the app is already deployed, that's a code-side config change, and you need to re-export and redeploy the jar for it to take effect — platform-side changes alone won't propagate into an already-running app.

## The Full End-to-End Flow (Recap)

Pulling together everything from this chapter and the previous one, here is the complete path from an idea to a secured, running API, as it was recapped in the course:

1. Design the RAML in **Design Center**.
2. Publish the RAML to **Exchange**.
3. Import it into **API Manager**, where an API instance is auto-generated.
4. Apply a **policy** in API Manager, based on the organization's security standard (Basic Authentication or Client ID Enforcement — never both at once).
5. Configure **Auto Discovery** (with the API ID) in the Mule app in Studio, linking it to this managed API instance.
6. Deploy the application to **Runtime Manager**.
7. Configure the environment-level **Client ID / Client Secret** (`anypoint.platform.client_id` / `anypoint.platform.client_secret`) in the app's Runtime Manager properties — only then does the API move to **Active** status and the policy actually get enforced.

One caveat worth remembering: editing the RAML/API definition in Design Center after this point does **not** automatically ripple through to an already-deployed application. Design Center changes are, at that stage, for documentation/design purposes — you'd need to re-import and redeploy for runtime behavior to actually change.

## Why Security Matters Beyond Just Restricting Data Format

It's worth stepping back and connecting this to Chapter 12's point about restrictions. A RAML-only API can already reject the wrong method or the wrong media type — but it will still process a perfectly well-formed request from *anyone*, including systems or people who have no business calling it. Restrictions answer "is this request shaped correctly?" Security answers "should this caller be allowed in at all?" Those are genuinely different questions, and MuleSoft deliberately separates them into two different layers: RAML/API Kit for shape, API Manager policies for identity. A production API almost always needs both.

## Common Mistakes / Things Students Got Wrong

- **Confusing the two "client ID/secret" pairs.** The single most common point of confusion in this material was mixing up the policy-level Client ID Enforcement credentials (what the caller sends) with the Access Management–level platform credentials (`anypoint.platform.client_id`/`client_secret`, what your app presents to the platform). They look similar and are both called "client ID/secret," but they solve different problems and live in different places.
- **A misspelled property key.** A small typo in `anypoint.platform.client_id` or `anypoint.platform.client_secret` in Runtime Manager's Properties screen will silently prevent the app from deploying correctly — this was called out explicitly as a real failure students ran into.
- **Deploying without the platform properties first.** The first deploy attempt after applying a policy commonly failed because the Client ID/Secret properties hadn't been added yet — the fix was adding them (from Access Management) and redeploying.
- **Putting Custom Expression credentials in the wrong Postman tab.** When a Client ID Enforcement policy is configured with the Custom Expression option (headers), putting `client_id`/`client_secret` in Postman's Authorization tab instead of the Headers tab throws an unauthorized error — they need to go in Headers to match `attributes.headers.clientId`.
- **Expecting a Design Center edit to affect a live app.** Editing the API definition in Design Center after deployment does not automatically update the running application — the app must be redeployed for changes to take effect.

## Quick Recap

- RAML-based restrictions (Chapter 12) control *shape*; API Manager policies control *identity/security* — they are separate layers.
- **Auto Discovery** links a running Runtime Manager app to its managed API in API Manager, using the API ID; policies only take effect on deployed (CloudHub) apps, never on local runs.
- **Basic Authentication**: username/password entered directly into the policy config in API Manager. No credentials from Postman → 401.
- **Client ID Enforcement**: credentials are auto-generated in Exchange (Request Access → Create New Client App), not typed manually. More secure than Basic Auth because credentials are longer, random, and permanent once created.
- Only one policy can be active per API at a time.
- Client ID Enforcement supports two credential-passing modes: Custom Expression (separate headers, default) or HTTP Basic Auth header (client ID as username, secret as password).
- Two distinct credential pairs exist: **policy credentials** (what the caller sends) vs. **Access Management/platform credentials** (`anypoint.platform.client_id`/`client_secret`, what the deployed app presents to the platform itself).
- Full flow: design RAML → publish to Exchange → import to API Manager → apply policy → configure Auto Discovery → deploy → set platform client ID/secret properties → API goes Active.
- Test secured APIs in Postman by replacing `localhost` with the CloudHub URL and supplying credentials in the format the policy expects; no credentials should return 401.
