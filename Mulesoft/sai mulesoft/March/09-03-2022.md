## Choice Router and Variables
- Extract a value from the payload (e.g. `payload.pho.brand`) and assign it to a variable using Set Variable, rather than referencing the raw payload path directly inside a Choice router condition.
- In the Choice condition, reference the variable using `var.<variableName>` (e.g. `var.brand`) — keeps conditions short and readable, especially with deeply nested payloads or payloads with multiple root tags.
- Example condition: `var.brand == "oneplus"` — if true, the flow enters that branch; if false, it falls through to the default route.
- Reasoning given: if you reference a long payload path directly inside a condition it becomes unreadable, especially for large/complex payloads — storing the value in a variable first keeps the choice condition clean.

## Scatter-Gather Router for Multi-Target Routing
- Used when a satisfied condition (e.g. brand = oneplus) needs to send data to multiple target systems at once.
- Example setup with two targets inside the Scatter-Gather:
  - Target 1 (Order): accepts JSON directly — no Transform Message needed, source data passed as-is.
  - Target 2 (Shipment): accepts XML — needs a Transform Message to convert JSON to XML before sending.
- Loggers were added after each target call for tracking purposes.
- For a different resource where the source data is already JSON and the target also expects JSON, an existing (redundant) Transform Message step was deleted.
- If the Choice condition is not satisfied, the flow routes to a different branch that uses a Transform Message to convert the data as required for that alternate target.

## API Kit Router (Auto-Routing by Resource)
- APIKit Router automatically routes incoming requests to the correct flow based on the resource name defined in the API spec — no manual connectors are needed inside the flow itself.
- Example: requests for different resources (e.g. track/trail) automatically route to their respective flows.
- A Listener should not be manually configured on a flow that is already invoked via the APIKit Router — the router itself receives and routes the request.

## Client ID Enforcement Policy & Runtime Properties
- When applying a policy that requires Auto Discovery, the runtime properties must declare:
  - `anypoint.platform.client_id`
  - `anypoint.platform.client_secret`
- Declaring these properties is what allows the API to show as "active" in API Manager, and is required for the application to deploy and link correctly to API Manager.

## Reading XML Data via DataWeave (Java Format)
- XML payloads cannot be read directly with a simple field path in Set Variable — doing so causes a namespace issue.
- Workaround: in Transform Message, set the output to `application/java` first, then access fields from the resulting Java object (e.g. `payload.phone.brand`).
- This converts the XML into a Java representation that can then be safely navigated for further processing (e.g. in a Choice condition).

## Logging — Debug vs Info Levels
- Loggers exist purely for tracking/troubleshooting — removing them does not affect the actual data delivery to the target system, only the ability to trace what happened.
- Two logging levels are used together in the same flow:
  - **Debug level**: log the complete payload here (can be large). Debug logs are not printed or stored unless the logging category is explicitly enabled in Runtime Manager.
  - **Info level**: log only a small, unique identifier (e.g. order ID, shipment ID, trailer ID) — this is always printed and used for tracking without consuming excessive log storage.
- Logging category naming convention: `com.<companyName>.<flowName>.<targetName>` (e.g. `com.logging.loggingflow.target`) — flexible, but should follow the organization's own standard.
- To enable/view debug logs: Runtime Manager → application → Settings → Logging → select Debug level and enter the category name → Apply Changes. Use the same path to disable it again once the issue is resolved.
- Troubleshooting workflow with debug logs: enable debug logging for the category, ask the source system to re-trigger the request, inspect the full payload in the logs, then disable debug logging again afterward.
- Practice pattern demonstrated: store the incoming unique ID (e.g. order number) in a flow variable near the start of the flow, then reference that variable (`var.order`) in every info-level logger throughout the flow, including a final "flow completed" log — this allows the same request to be traced end-to-end using just the unique ID.
- Log storage limits:
  - **Runtime logs** (Runtime Manager): max ~100 MB, retained for up to 30 days.
  - **Anypoint Monitoring**: up to 90 days of backup, with no fixed storage size limit — better suited for larger volumes of historical data.
- Two ways to check logs: application-level logs in Runtime Manager, or Anypoint Monitoring.
- There is no single universal rule for logging standards — it depends on the organization (one past client, Unilever, enforced very strict logging standards with architect code review). The debug-for-full-payload / info-for-unique-ID pattern was presented as a solid default approach to follow.
