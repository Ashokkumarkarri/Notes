## JDA-to-Pink Integration — Real-Time Project Walkthrough
- Example real-time project: a Yard Management System integration between two systems, **JDA** (source) and **Pink** (target).
- The project deals with different material types moving through a yard — trailers, containers, trucks — tracked as they check in/out of the yard.
- JDA and Pink cannot communicate directly, so Mule sits in between as the integration layer.
- Pink only accepts data in **JSON** format; JDA sends data in **XML**, so Mule transforms XML → JSON before forwarding.
- HTTP method used: **POST**, because JDA is triggering/pushing data into Mule (not retrieving). GET is used only when retrieving data from a target system.
- Multiple resources were covered, each following the same pattern of logic:
  - **Appointment** — resource endpoint for appointment data (Seal, Appointment, Location, XML).
  - **Trailer / Checkout** — triggered when the warehouse records a trailer/truck checking out; uses an **Operation Code** field to distinguish between values like `CHK OUT` (checkout) and `CHK IN` (check-in); if no operation code is found, it's logged separately.
  - **Work Queue** — another resource following the same operation-code-based branching logic.

## API Naming Convention
- API naming pattern used: `com.<company-name>.<source>-to-<target>` (e.g., `com.<company>.jda-to-pink`), written in a readable, categorized manner so the source/target pairing is clear from the name alone.

## Logging & Tracking Strategy
- **Outbound Segment Number** is the unique identifier present in the data coming from JDA (source session) — used as the tracking key.
  - Whenever a transaction fails, the business team can provide this outbound segment number, and it can be searched in the logs to trace the failure.
  - Two places to check logs: (1) application/runtime logs, and (2) **Anypoint Monitoring**, which retains **90 days** of log backup.
- Standard logging pattern applied at each resource/flow:
  1. Log the **received payload** (raw XML from JDA) — logged at **debug** level.
  2. Set variables for tracking — e.g., **Outbound Segment Number** and, where applicable, **IB flag value** / **Operation Code**.
  3. Log the **Outbound Segment Number** — logged at **info** level (category not mandatory for info level, but kept as a standard practice so it's always visible in runtime logs).
  4. After transforming XML to JSON, log the **transformed payload** — also at debug level.
  5. Log a confirmation message such as "Connection initiated to Pink" and, after the call succeeds, "Successfully triggered to Pink" — if this final logger doesn't print, it indicates the target system didn't respond or a component failed and execution went into exception handling.
- A single logger can print multiple variables/values together (e.g., both a variable and a payload) — this was demonstrated as reusable practice from an earlier day's task.
- Category is not mandatory for info-level logs; if no category is set, the log still prints by default. This is treated as the team's logging standard.

## Flow Components
- **Choice Router**: used to branch logic based on conditions provided by the business team. Example: an **IB flag value** in the source data can be `01` or `02`; each value routes to different logic. If the value is neither `01` nor `02`, execution falls through to the **default condition** (i.e., condition not satisfied).
- **Flow Reference Component**: used to call/reference another flow directly from the main flow.
- **Amazon SNS Connector**: used specifically for connecting to and sending data to Pink.
- Component- and variable-naming conventions must be kept clear and consistent across flows (e.g., naming variables to reflect their purpose like outbound segment number, operation code).
- The overall structure — log received payload → set tracking variables → transform payload → log transformed payload → flow reference/send to target → log success — is repeated consistently across all resources (Appointment, Trailer/Checkout, Work Queue), described as "the entire application will be a replica" of this same pattern.

## Naming Conventions & Deployment
- Environment/application naming example: `E-Commerce-Production-Sandbox-Dev`.
- **Dev** refers to the Sandbox environment; **Production** refers to the live Production environment.
- Deployment workflow:
  1. While an application is under testing, deploy it to **Dev** (Sandbox).
  2. Once the business team tests it and signs off confirming it works as expected, the code is moved to **Production**.
  3. Moving to production is done by uploading the same application file to the production environment.
