## API Policies: Client ID Enforcement vs Basic Authentication
- Two policy types can secure an API in API Manager: **Client ID Enforcement** and **Basic Authentication**.
- Only **one** policy can be applied to a single API at a time — never both together. Which one to use depends on the organization's own security standards; there's no fixed rule.
- **Client ID Enforcement** is considered more secure than **Basic Authentication**:
  - Basic Authentication credentials are a username/password you create yourself — simple, and could be reused for a long time (e.g., kept unchanged for many days).
  - Client ID Enforcement credentials (client ID + client secret) are auto-generated by the backend system, are much longer strings, and **cannot be changed** once generated — only created once.
- Client ID Enforcement policy supports two ways of passing credentials:
  - **Custom Expression** (default): client ID and client secret are passed as separate headers. The default expression referenced is `attributes.headers.clientId`.
  - **HTTP Basic Authentication header**: client ID is passed as the username and client secret as the password, sent via the Authorization header.
- Steps to configure and apply the policy:
  1. In API Manager: **Manage API → Manage API from Exchange**, search by the API name (e.g., "Amazon").
  2. If the API was only designed in Design Center but never published, first publish it: **Design Center → open API → Publish to Exchange**.
  3. Back in API Manager: **Import from Exchange → Save**.
  4. Go to **Policies → Add Policy → Client ID Enforcement**.
  5. Choose either the Custom Expression option or the HTTP Basic Authentication header option, then **Apply**.
  6. A policy can be applied either before or after deploying the application — order doesn't matter.
- Testing in Postman:
  - With Custom Expression: pass `client_id` and `client_secret` as separate entries under **Headers** (not in the Authorization tab). Putting them in Authorization instead throws an "unauthorized" error.
  - With the HTTP Basic Auth option: use Postman's Basic Auth tab — username = client ID, password = client secret.
  - Other REST client tools (SoapUI, ARC – Advanced REST Client) behave the same structurally (same headers, same setup) as Postman.

## Publishing to Exchange & Generating Client ID / Secret
- Applying the Client ID Enforcement policy in API Manager only turns the policy **on** — it does **not** generate the client ID/secret. Those credentials must be generated separately in Exchange.
- Steps to generate credentials:
  1. Go to **Exchange** → open the published API (e.g., Amazon) → **Request Access**.
  2. Specify the API ID (visible in API Manager for that API).
  3. Check the Application dropdown for existing access; if no client app exists yet, create one via **Create New Client App**.
  4. Once created, open the application to view its auto-generated **Client ID** and **Client Secret** — copy these for use in Postman / the source system.
- For **Basic Authentication**, by contrast, the username and password are entered directly in API Manager when applying the policy — no separate Exchange step is needed.
- For the app to move into **Active** status in API Manager (policy actually enforced), the application must also be deployed in Runtime Manager, supplying the environment's client ID and secret so it can be validated via Access Management.

## HTTP Request Connector
- Use the **HTTP Request** connector whenever connecting to any API-based target system, because APIs are exposed over the HTTP/HTTPS protocol.
- Configure: protocol, host, path, and method (e.g., POST).
- If the target system is secured with a policy, also configure the matching credentials inside the HTTP Request connector.

## Doubts / Q&A
- **Q: If you use a different testing tool instead of Postman, is the setup the same?**
  A: Yes. Common REST testing tools are Postman, SoapUI, and ARC (Advanced REST Client). Any of them can be used — they all have the same structure (headers, etc.).
- **Q: Which is better — Client ID Enforcement or Basic Authentication?**
  A: Client ID Enforcement is more secure. Basic Authentication uses simple, self-created username/password credentials that could persist unchanged for a long time. Client ID Enforcement credentials are auto-generated by the backend, are a long random string, and cannot be modified once created — only generated once.
- **Q: We've only worked with JSON and XML so far — will the remaining data structures/media types be covered too?**
  A: Yes, they will be covered in upcoming sessions (formats worked so far also include Java, CSV, and text via file connectors).
