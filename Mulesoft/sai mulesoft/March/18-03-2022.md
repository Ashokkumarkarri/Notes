## Debugging in Anypoint Studio
- Live troubleshooting session on a student's application that had an issue.
- Steps used to debug: open **Global Elements**, check the relevant configuration, review **runtime logs**.
- To debug locally:
  1. Go to the **main flow** (not the exception handling flow) and add a **breakpoint**.
  2. Run the application in **Debug mode** in Anypoint Studio.
  3. Step through execution to identify exactly where the flow is failing.
- General guidance: in real projects, once an application is confirmed working in a non-production environment and signed off by business, it is moved to production. If an unexpected issue occurs, it should first be debugged locally (breakpoints + debug mode) to isolate the failing component.
- When copying error messages for troubleshooting, copy the entire error text (Ctrl+A, Ctrl+C) into Notepad for review.
- Observed error in this session: a "too many nested child contexts" issue traced back to the **Correlation ID Generator** configuration in Global Elements — removing/deleting that configuration component resolved it.

## Correlation ID Generator Issue
- Global Elements contained a Correlation ID Generator configuration causing the nested-context error.
- Fix: delete the existing configuration component, then re-create a fresh configuration (Create → search "Configuration" → drop it in → Save) rather than reusing the broken one.
- Note on environment/backend IDs: every environment has its own ID (sandbox ID, design ID, production ID, non-prod ID, pre-prod ID, etc.). In the trial/sandbox environment used for training this ID isn't visible, but in real time this ID must be configured and is provided by the platform admin on request.

## Exception Handling & Logging Best Practices
- If an issue can't be identified from runtime logs, the root cause is usually insufficient logging — add loggers not just in the main flow but also **inside the exception handling part**.
- Recommended: add a logger in the **On Error Propagate** block (or wherever HTTP errors are caught) to record which specific error occurred — e.g., whether it went into an **HTTP CONNECT** block, a **timeout** block, or an **HTTP NOT FOUND** block.
- In the exception handling logger, print the unique tracking identifier (e.g., the outbound segment number) so failures can be traced back to a specific transaction.
- If the application's final success logger doesn't print in the logs, that's the signal something failed before reaching the target system.

## Email Notifications: CloudHub Notifications vs SMTP Connector
- Two ways to trigger an email alert when a flow has an issue: the **CloudHub Notifications connector** and an **SMTP connector**.
- Important constraint: the **CloudHub Notifications connector only accepts payload in text/plain format** — it will not accept JSON directly.
  - If the current payload is JSON, it must be read/converted to text format (e.g., using a `read` operation) before sending it through the CloudHub Notifications connector, otherwise the email will not trigger.
  - This text-only limitation is specific to the CloudHub Notifications connector.
- The **SMTP connector** does not have this limitation — it can trigger notifications with JSON payloads directly.
- Recommendation given: CloudHub Notifications connector is considered the preferred/best approach for triggering these alerts in this course's context, despite the text-format requirement.

## Anypoint Platform Environments & Business Groups
- Anypoint Platform environments discussed: **Design**, **Sandbox**, and **Production** (three environment types by default).
- In a real project, typical environment naming looks like: `ecommerce-dev`, `ecommerce-prod`, `ecommerce-pre-prod`.
- Environments can be renamed (e.g., renaming "Design" to a custom name) but the trial account only allows a **delete** option for some environments — no edit option is available for certain ones.
- **Business Groups**: a business group can contain multiple environments (default environments like Sandbox and Design exist inside a business group). In real projects, each business group typically maps to one organizational branch/unit, and there can be multiple business groups within an organization.
- A student's clarifying question about "what is a business group" was answered: business groups are used to logically separate environments/applications, and each business group has its own set of environments (e.g., Sandbox, Design) under it.

## Moving Applications Between Environments
- On the trial account, there's no direct "move application" action between environments — applications must be moved manually.
- Manual process to migrate an application to a new environment:
  1. Go to **Runtime Manager** → open the application → go to **Settings**.
  2. Download the application's **JAR file** to the local machine (clicking the application name under settings triggers the download).
  3. Repeat this for all applications that need to move.
  4. Delete the old environment (e.g., Sandbox) once all applications are downloaded.
  5. Upload the downloaded application JAR files into the new environment.

## Task: XML-to-Target Field Mapping (Order / Shipment API)
- Assigned task: create an API with **one resource**, method **POST**, that accepts **XML** data as input.
- Sample source data contains a large number of fields (referenced at different points as **29 fields** and **33 fields** across the two example payloads/resources) — but **not all fields are required by the target system**.
- Requirement: extract only the mandatory subset of fields needed by the target (e.g., only the **first 10 fields** in one example) and send only those onward — do not forward the entire source payload.
- Two related URLs mentioned as active/available for the target application: a **Shipment** URL and an **Order** URL.
- Naming convention noted for these values: the first letter should be capitalized (e.g., `Order`, `Shipment`).
