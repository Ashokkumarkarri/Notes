## Ways to Handle Exceptions in Mule 4
- Exceptions in Mule 4 can be handled in three ways:
  - **Flow level** — an error handler scoped to a single flow.
  - **Global level** — a shared error handler applied across the whole application.
  - **Process level** — using a **Try** scope around a specific set of processors inside a flow.
- Key error-handling components in the Mule Palette: **On Error Propagate**, **Error Handler**, **Raise Error**, **Try** (process-level scope), and a **Global Error Handler** (global level).
- If an error type is not explicitly declared on a block, it defaults to **ANY**.

## Building/Importing a RAML API Manually
- Alternate way to bring an API into Studio besides the normal Design Center → download/sync flow:
  1. Close unrelated/unused projects in Studio first.
  2. Under `src/main/resources`, create a new file with a `.raml` extension (e.g., `AmazonAPI.raml`).
  3. Copy the RAML content from Design Center and paste it into this local file.
  4. Right-click the RAML file → **Mule → Generate flows from local REST API** to generate the APIKit router flow.
  5. This manual/local approach is **not linked back** to Design Center — changes made in Design Center won't sync automatically.
  6. Alternative: save the RAML file to the desktop and import it into the project locally.
- What's visible in a basic RAML file at this stage is only the **skeleton** (structure). A more advanced RAML topic — **fragments** — was mentioned but deferred to a later session (to avoid confusion before the skeleton/structure basics are solid).
- The sample API used had two resources in its RAML: `order` and `shipment`. The `shipment` resource/flow was removed for this exercise, keeping only `order` for the flow-level error-handling demo (re-added in the next file for the global-handling demo).

## Flow-Level Error Handling Demo
- Demo flow built on the `order` resource:
  1. **Logger** (INFO level) — logs the incoming payload as soon as it's received.
  2. **Transform Message** — converts the payload from JSON (as received from the source system) into XML.
  3. **Logger** — logs a "request sending to target" message before calling the target system.
  4. **HTTP Request** to the target system, deliberately configured with invalid credentials/invalid path to force a failure.
  5. **Logger** after the HTTP request — only reached if the call to target succeeds, logging a success message.
- Reiterated logging rule: log the full payload only at **DEBUG** level, not INFO.
- Flow-level error handling: an **On Error Propagate** component was dropped onto the flow with a custom error message ("issue with the target system"). The error type field was left as default, which is treated as **ANY** if not explicitly declared.
- To link a flow's unhandled errors to a **Global Error Handler**: in **Global Elements**, add/select the global error-handling configuration and associate it with the main flow, so any error not caught locally routes to the global handler automatically.

## Global-Level Error Handling Demo
- Both the `order` and `shipment` flows were configured with invalid target credentials so their requests would intentionally fail.
- When the `shipment` flow's request failed, it went to the **Global Error Handler** (not a flow-level handler), confirming that global handling catches errors from multiple flows at once.
- Even within global-level handling, if different error scenarios need different behavior, each error type still needs its **own dedicated block** — the global handler is evaluated top-down just like a flow-level handler.
- Concrete error types configured as separate blocks in the global handler for this demo:
  1. `HTTP:TIMEOUT`
  2. A second HTTP-related error type (connectivity-style failure)
  3. `HTTP:UNSUPPORTED_MEDIA_TYPE` — for when the target system responds with the wrong/unsupported media type
- Each of the three blocks contained its own **Transform Message**, so behavior could differ per error type even though all three live under one global handler.
- Trade-off noted: global-level handling centralizes error handling ("only one person is handling it") across all flows. If a specific flow needs its own distinct behavior for certain error types, it's better to keep a flow-level handler for those specific types instead of relying solely on the global one.

## Doubts / Q&A
- **Q: If the target system accepts JSON from some resources but XML (or other formats) from others, how do you configure error handling globally to cover all of them?**
  A: Global error handling is still defined **per error type** — you add a separate error-propagate block for each specific error type inside the Global Error Handler (rather than one catch-all block), so different resources/scenarios can be handled distinctly even though they share a single global handler.
