## Securing HTTP Request Calls with Basic Authentication
- The demo target application from the previous session is secured with a **Basic Authentication** policy.
- Calling it without credentials returns a **401 Unauthorized** error.
- Fix: in the HTTP Request connector, click **Edit configuration** → scroll down to **Authentication** → select **Basic Authentication** → enter the username and password.
- Demo credentials: username = `target`, password = `target@123` (note: the `T` must be capitalized).
- After configuring authentication, the request against the secured target endpoint succeeds.

## Choice Router — Condition-Based Routing
- **Choice Router** is used whenever data should only be sent to the target system if a business condition is true.
- Example conditions discussed:
  - Only send data to target if brand value = **"OnePlus"**.
  - Only send data to target if color = **"black"**.
- If the condition evaluates true, that route executes (e.g., sends data to target).
- If false, execution falls into the Choice Router's **default** route. In the demo, the default route had no HTTP Request connector, so the message is simply not sent to target and the flow ends there (the message is effectively suppressed).
- Conditions reference the payload structure directly, e.g. `payload.phone.brand`, where `phone` is the XML root tag and `brand` is a child element — compared for equality against a value like `"OnePlus"` or `"Samsung"`.
- A single Choice Router can hold any number of conditions/routes, each pointing at different downstream logic (e.g., different target resource/endpoint per brand).
- Demo walkthrough:
  - Request with brand = OnePlus → condition true → routed to the **order** endpoint → target responded "Order placed successfully."
  - Request with brand = Samsung → initially fell into the default route (no matching condition existed yet) → instructor added a second condition/route for Samsung, pointing to the **shipment** endpoint (different URL) → after adding it, Samsung requests routed correctly.
- For the shipment resource, the target expects **XML**, but the data arriving was JSON — so a **Transform Message** step was added before the HTTP Request to convert JSON → XML.

## Practical Hands-On Task Assigned
- Build an API with two resources: **salesforce** and **trailer**.
- **salesforce** resource: accepts XML input → transform (Transform Message) to JSON → send to the target's **order** resource.
- **trailer** resource: accepts JSON input → transform to XML → send to the target's **shipment** resource.
- Both flows require a Choice Router condition: only forward data to target if **color = "black"**; otherwise log "condition not satisfied" / "color does not match" and do not send.
- Target application is secured with Basic Authentication — username: `target`, password: `target@123` (capital `T`).
- Add proper logging for both the success path and the "condition not satisfied" path.

## Doubts / Q&A
- **Q: Why is my request to the target system failing?**
  A: The student's HTTP Request path ended in `/api/salesforce`, but the target application only exposes two valid resources — **order** and **shipment**. Any other resource name in the path returns a **404 Not Found**. The instructor walked through checking the failing request's URL in the console/logs to confirm the path mismatch, then corrected it to `/api/order`.
