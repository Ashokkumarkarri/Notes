## Systems Involved in a Mule Integration Flow
- A typical integration has three systems: the **source system**, the **Mule application**, and the **target system**.
- The source system triggers/sends data into Mule (in the demo, Postman played the role of the source system).
- The Mule app receives, optionally transforms, and forwards data to the target system.
- **HTTP Listener** connects the source system to the Mule app — you expose an endpoint (URL) in the listener and hand that endpoint to the source system so it can trigger data into your flow.
- **HTTP Request** connects the Mule app to another (target) system — used whenever Mule needs to send data out to another API-based system.
- Both HTTP Listener and HTTP Request are referred to as **connectors**.
- Rule of thumb: any time Mule talks to an API-based system, an HTTP connector is involved — Listener to *receive* from a system, Request to *send* to a system.
- Once the request reaches the Listener, it goes to the **APIkit Router**, which routes to the correct flow based on the resource name (e.g., a request for the "shipment" resource routes to the shipment flow).

## Endpoint Structure for API-Based Systems
- An endpoint for an API-based system has the format: `protocol://host:port/path`
- In the HTTP Request connector, you configure: **Host**, **Port**, **Protocol**, **Method**, and **Path**.
- The Method must match what the target resource supports (e.g., if the resource accepts POST, configure the request's method as POST).
- The Path must match the exact resource path exposed by the target application.

## Demo — Building a Target Application in Anypoint Studio
- The instructor built a separate sample Mule app purely to simulate an external "target" system, with two flows: **order flow** and **shipment flow**.
- The API definition (RAML) lives under `src/main/resources`.
- **Order** resource: accepts POST, media type **JSON**.
- **Shipment** resource: accepts POST, media type **XML**.
- After deploying this app, its two endpoints were:
  - `http://<host>:<port>/api/order`
  - `http://<host>:<port>/api/shipment`
  - (the `*` in the RAML/API path is a wildcard covering all resources under `/api`)
- Even though this "target app" is technically another Mule application, it's meant to be treated as if it were an external system (e.g., imagine it as Salesforce or some other vendor system).

## Live Demo — Connecting Source → Mule → Target
- Built a new Mule app with:
  - HTTP Listener (receives data; source = Postman)
  - Logger to print the incoming payload
  - HTTP Request (sends data to the target app's **order** endpoint, method POST, path `/api/order`)
  - Logger to print the response payload
- Ran the app in **debug mode**, stepping through as requests were triggered from Postman.
- **Test 1:** sent plain text data to the order endpoint (which expects JSON) → request failed with a media-type validation error, because the target app validates incoming data and only accepts JSON for the order resource.
- **Test 2:** sent proper JSON data → request succeeded; target responded with **"Order placed successfully."**
- Key takeaway: after the HTTP Request step executes, the Mule payload gets **overwritten by the response** coming back from the target system — the original inbound payload is not automatically preserved past that point.

## Debugging — Reading a 404 Error
- To demonstrate an error scenario, the instructor deliberately configured the HTTP Request with a resource name/path that doesn't exist on the target system.
- Result: the request failed with a **404** status code, meaning "resource not found" — the target application has no such resource at that path.
- When a component in the flow errors out, execution can stop at that point, but a logger placed to print the final response (including error responses) still shows what came back from the target system in the console/logs.
