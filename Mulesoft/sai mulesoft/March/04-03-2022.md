## Building the Two-Resource API (Salesforce / Trailer)
- Created a new API spec named **database-api** with two resources:
  - **salesforce** — accepts XML
  - **trailer** — accepts JSON
- Created a new Mule application and imported the RAML into Anypoint Studio.
- When an API is imported with an **APIkit Router**, Studio auto-generates one flow per resource declared in the RAML, plus the router flow and an error-handling flow — in this case, 3 flows total (main router + salesforce + trailer).

## Salesforce → Order Flow
- Logger prints "request received from Postman" along with the incoming payload.
- Incoming data for the **salesforce** resource is XML.
- The target's **order** resource expects JSON, so a **Transform Message** step converts XML → JSON before the HTTP Request.
- A logger prints the transformed payload before it's sent to the target.
- HTTP Request calls the target host at `/api/order`, method **POST**.
- A logger prints the response payload received back from the target.
- A **Choice Router** condition checks `payload.phone.color == "black"` — only when true is the data sent to the target's order endpoint; otherwise it logs "condition not satisfied" and the message is not forwarded.
- Debug walkthrough confirmed the full chain: XML in → transformed to JSON → sent to target → target responds with a success message, and all three loggers (raw payload, transformed payload, target response) print correctly in sequence in the console.

## Trailer → Shipment Flow
- Same pattern, mirrored: incoming data for **trailer** is JSON, but the target's **shipment** resource expects XML.
- A **Transform Message** step converts JSON → XML before sending.
- The same "color = black" Choice Router condition is applied before forwarding to target.
- HTTP Request calls the target host at `/api/shipment` (same host as order, only the path differs), method **POST**.
- Debug walkthrough confirmed correct transformation and conditional routing, matching the salesforce/order flow.
- Note: even without manually placing a breakpoint, running the app in **Debug Mode** in Anypoint Studio lets you step component-by-component to verify the payload at each stage of the flow.

## Securing the API with Basic Authentication (Anypoint Platform)
- In Anypoint Platform, the API (**database-api**) is managed from Exchange in **API Manager**, which provides an API ID.
- In Studio, an **API Autodiscovery** global element is added to the main flow using that API ID — this links the running app to the managed API definition on the platform.
- In API Manager, under **Policies → Apply Policy**, search for **Basic Authentication** and configure it with a username/password.
  - Demo credentials used at this stage: username `viresh`, password `viresh@123` (capital letters matter).
- The application is then deployed to **CloudHub** via **Runtime Manager → Deploy App**.

## CloudHub Deployment — Client ID/Secret & Troubleshooting
- The first deploy attempt failed: an app with a platform policy applied (like Basic Auth via API Manager) needs the **Client ID** and **Client Secret** configured as app properties before it can start successfully.
- Client ID and Client Secret are obtained from **Access Management** for the relevant environment (e.g., Sandbox).
- In Runtime Manager, open the app → **Settings → Properties**, and add:
  - `anypoint.platform.client_id`
  - `anypoint.platform.client_secret`
- After adding these properties and redeploying, the app starts successfully and its status shows **Active** in API Manager.
- When calling the CloudHub-deployed app, the `localhost`/local port used during local testing must be replaced with the app's actual CloudHub URL.
- Testing without credentials returned **401 Unauthorized** (Basic Auth policy is enforced); adding Basic Auth (username/password) in the client returned **200 OK**.
- Instructor noted that policies can also be applied to an app **after** it has already been deployed, not only before.

## Doubts / Q&A
- **Q: Why didn't the app deploy successfully the first time after applying the Basic Authentication policy?**
  A: An app secured by a platform policy needs the Anypoint platform **Client ID** and **Client Secret** set as properties (`anypoint.platform.client_id` / `anypoint.platform.client_secret`) in Runtime Manager, fetched from Access Management. Without them the app fails to start / goes to an undeployed state. Adding the properties and redeploying resolved it.
- **Q: Why is a "trailer" request not reaching the target system even though color looks correct?**
  A: On checking the runtime logs, the payload's color value was actually `"blac"` (missing the final "k") rather than `"black"` — an exact string mismatch, not a bug in the flow. Since the Choice Router condition requires an exact match to `"black"`, the request correctly fell through as "condition not satisfied" and was suppressed as designed.
