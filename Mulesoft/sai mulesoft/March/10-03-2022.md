## Client ID and Secret: Policy-Level vs Access Management-Level
- Two distinct types of client ID/secret used in Mule projects:
  1. **API-level (Client ID Enforcement) policy**, applied in API Manager — the source system must pass this client ID/secret every time it triggers data to the Mule app. These are policy credentials.
  2. **Access Management-level** client ID/secret — used to link a Runtime Manager deployment with API Manager (via Auto Discovery), so the API shows as "active." Declared in the app's runtime properties as `anypoint.platform.client_id` and `anypoint.platform.client_secret`.
- Client IDs at the environment (Access Management) level always differ between environments (e.g. Sandbox vs another environment); Platform-level IDs are unique/global to the organization.
- Environments (Anypoint Platform → Environments) can be named anything — there is no rule that one must be called "Sandbox." Real projects commonly use names like UAT, Production, Non-prod.
- To retrieve a client ID/secret: Access Management → Environments → select the environment → copy the client ID and secret (e.g. into Notepad for later use in runtime properties).

## Vertical Scaling (vCore) vs Horizontal Scaling (Workers)
- **Vertical scaling (vCore)**: increases the compute/memory allocated to a single worker. Used when a single request carries a huge payload. Default allocation is 0.1 vCore; can be scaled up toward 32 GB. Higher vCore allocations cost significantly more — large allocations (16GB/32GB) can run into crores of rupees yearly, since it consumes Mulesoft platform resources the client is billed for.
- **Horizontal scaling (Workers)**: increases the number of workers, where each worker is effectively a separate server instance (e.g. one located in Ireland, another in the UK/Australia — no two workers run in the same location). Used when the volume of incoming requests is high, regardless of individual payload size. Workers are numbered starting from 0 (1 worker by default); adding more workers spreads load across multiple servers (e.g. of 1000 incoming requests, roughly half could be handled by one worker/server, the rest by another).
- Rule of thumb: vCore (vertical) → single request with a huge payload; Workers (horizontal) → high number of concurrent/total requests.
- Efficient application design should avoid over-provisioning unused vCore or workers, since both directly increase cost. Teams are expected to design flexible, resource-efficient applications within the budget set by the client/organization.
