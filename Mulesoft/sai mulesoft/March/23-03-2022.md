## Recap: API Types & Mule 4 Message Structure
- Two API types reviewed: **REST** and **SOAP**.
- Mule 4 message structure: **Payload**, **Attributes**, and **Variables**. Mule 4 has only **one** type of variable.
- Contrast with Mule 3: Mule 3 used **Properties** (not Attributes) and had **three** variable types — **Flow variable**, **Session variable**, and **Record variable**.

## Recap: Routers
- Three router types were reviewed as part of the recap; **Choice router** and **Scatter-Gather router** were explicitly named. (A third router type was referenced in the discussion but the exact name was unclear in the recording.)

## Recap: Loggers & Scaling
- Loggers are used for **tracking** and **printing** — to print logs, payload data, or any unique value needed for troubleshooting.
- Two log levels covered: **Info** and **Debug**.
  - Info-level logs print automatically by default in runtime logs.
  - Debug-level logs require the debug category to be explicitly enabled to appear in runtime logs.
- Scaling in Mule 4: two types — **Vertical scaling** and **Horizontal scaling**.
  - Vertical scaling relates to **vCore** allocation.
  - Horizontal scaling relates to adding more **Workers**.

## Recap: Exception Handling
- Exceptions can be handled at two levels: **Flow level** and **Global level**.
- **Global level**: a separate global configuration (created under Global Elements/Configuration) defines the error-handling logic; the main flow references this global configuration so any unhandled error routes to it.
- **Flow level**: handled using flow-level exception components — **On Error Propagate**, **On Error Continue**, and a **Try** scope/block.

## Recap: Connectors
- Two connectors specifically reviewed: **HTTP Connector** and the **CloudHub notification-type connector**.
- **CloudHub notification connector**: used to trigger email alerts whenever there is an issue in the flow.
- **HTTP Connector** usage depends on direction:
  - If the **source** system connects into Mule and is API-based, the flow starts with an **HTTP Listener**.
  - If the **target** system is API-based, an **HTTP Request** is used to call out to it.

## Recap: Variables — Set Variable vs Transform Message
- Two ways to create a variable in Mule 4: the **Set Variable** component and **Transform Message**.
- Guidance on which to use:
  - For creating a **single** variable, prefer **Set Variable** — it's lighter and faster.
  - For creating **multiple** variables (e.g., 4–5 at once), **Transform Message** is more efficient to set them all in one place, even though Transform Message takes comparatively more time to execute than Set Variable.

## Task Review: XML Field Mapping to Warehouse ID
- Review of a task assigned in the previous (Friday) session: source system sends a large XML payload; only a few mandatory fields need to be sent to the target system.
- Sample XML had two root elements, with roughly 13 fields on one side and about 9 key-value fields on the other (from a check-in/check-out style payload with nested route/QC data).
- Target system in this example only required **one field** — a **Warehouse ID** — built from the source payload's route/QC data path; in the example shown, this field was hardcoded to a value of **990** as needed by the target.
- Key point emphasized: the source system can have any number/shape of fields, but the target's requirement dictates what actually needs to be extracted, transformed, or renamed (keys can be changed to match target naming) — this mapping flexibility is described as one of Mule's core strengths.
- General principle reinforced: know which component to use (logger, choice, variable, HTTP request, etc.) for a given requirement — the actual transformation logic itself is usually fast to build once the mapping is clear.

## MuleSoft Connectors & Establishing Connectivity
- MuleSoft has roughly **150–155 connectors** available in total; this course covers a practical subset of about **7–8** commonly used connectors.
- If a required connector isn't available in the Studio palette, it can be **imported from Anypoint Exchange**.
- Example given: if the target system is a specific type (e.g., a CRM-type system), use its dedicated connector from the palette and configure it (host, port, etc.) as required.
- Key real-world skill emphasized: before choosing a connector, analyze what kind of system the source and target actually are (API-based, database, file-based, etc.) — establishing correct connectivity between source and target is described as the main practical challenge faced on the job.

## Task: JSON Array Transformation (Employee Data)
- New task assigned: source system returns an **array of JSON** employee records (fields include employee name, salary, designation, location).
- Requirement: iterate the array and, for records where **designation = "Consultant"**, transform and send the following fields to the target system:
  - **Employee Name** → send the **character count** of the employee name (using a size/`sizeOf`-style function), not the name itself.
  - **Employee Salary** → send only the **first digit** of the salary value (e.g., a salary of one lakh or nine lakhs should be reduced to just its leading digit).
  - **Designation** → send as-is (same value).
  - **Location** → send only the **first three characters**, representing the location code (e.g., Bangalore → BAN, Hyderabad → HYD, Pune → PUN); same logic applies regardless of which city appears in the data.
- Suggested implementation approach: iterate the payload using **map** inside **Transform Message**, applying the above conditions per field for each object in the array.
- Logging pattern to follow: Listener → logger (print the received/raw payload) → Transform Message (apply the mapping logic) → logger (print the transformed payload being sent to the target).
- Task was described as short (10–15 minutes) and meant to be attempted individually before the next session.
