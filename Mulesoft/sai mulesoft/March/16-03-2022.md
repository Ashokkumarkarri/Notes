## Creating Multiple Variables via Transform Message
- Mule 4 has two ways to create a variable: **Set Variable** component, or a **Transform Message** with a DataWeave output.
- Rule of thumb: if you only need **one** variable, use Set Variable. If you need **multiple** variables, define them all inside a single Transform Message's DataWeave script instead of chaining many Set Variable components — this keeps the flow shorter and avoids unnecessary components.
- A single Transform Message can create many variables at once (mentioned up to 100) — no need to drag and drop a separate Set Variable for each one.
- When you drop a new Transform Message, its default output is `payload`. If you're only using it to set variables, remove the unneeded default payload output.
- Variable access syntax in Mule 4 is always `vars.variableName` (Mule 4 effectively has a single kind of variable — the flow variable).
- Demo: assigned the outbound segment number to a variable named `outboundSegment`, and the warehouse ID to a variable named `condition`, both set inside one Transform Message.
- Debugging: added a breakpoint at the Transform Message, ran in Debug mode, and checked the **Variables** panel to confirm both variables were assigned the correct values.
- General principle reiterated: reuse existing components rather than adding duplicate/unnecessary ones — this keeps flows shorter and easier to maintain.

## Why Proactive Failure Alerting Matters
- Scenario: Salesforce sends order data into Mule; Mule needs to forward it to a target system (a database), but the request fails due to connectivity issues with the target.
- Problem: without alerting, failures are only discovered when the user manually reports an issue or someone happens to check the logs. At real-world volumes (mentioned as 10,000–20,000+ requests/day), this doesn't scale.
- Solution: automatically trigger an email notification whenever a transaction fails in Mule, so the team is alerted before the user has to report it — without constantly checking logs.
- Notifications should be triggered **only on failure**, not on every success — triggering on every successful transaction (e.g., 10,000 times a day) would flood the mailbox for no benefit. (CloudHub Connector does support success notifications too, but there's little practical value since the target has already received the message successfully.)

## Setting Up the CloudHub Connector
- Tool used: **CloudHub Connector**, imported from Anypoint Exchange (Search in Exchange → search "cloudhub" → Add → Finish) since it isn't in the palette by default.
- Connector configuration requires:
  - Anypoint Platform **username and password**.
  - The target **environment** (e.g., "Sandbox").
  - A **domain name** that must exactly match the deployed application's name in Runtime Manager (e.g., "notification").
- Use **Test Connection** after entering credentials/environment to confirm the configuration is valid before proceeding.
- Recap of HTTP Request connector configuration for comparison: Host, Port, Path, Method, and (if the target is secured) authentication/security settings.

## Flow Reference — Sharing One Error-Handling Flow
- **Flow Reference** is the component used to call/invoke a separate flow from within the current one.
- Instead of duplicating the CloudHub Connector in every error block, each error type (`HTTP:TIMEOUT`, a connectivity-type error, and `ANY`) in the global error handler uses a **Flow Reference** to route to one shared flow — named the "error handling flow" — which contains the single CloudHub Connector used to trigger the notification.
- This reuse pattern avoids repeating the same notification logic across multiple error blocks.

## Configuring Anypoint Platform Alerts
- In Anypoint Platform Runtime Manager → **Alerts** → "Create your first alert."
- Alert configuration fields:
  - App name = the deployed application's name (e.g., "notification").
  - Recipients — email addresses to notify (limit of **20 email addresses** per alert).
  - Severity level — e.g., **Critical** means an emergency/priority issue where important users are affected and it needs immediate attention. Applications are generally categorized as critical or non-critical (based on business impact) to avoid unnecessary escalations for less important apps.
  - Condition — select the deployment target (CloudHub) and the specific application.
  - Alert type — **Custom Application Notification**, used because this is a custom, self-generated alert triggered from inside the Mule flow rather than a platform-default alert.
  - Message content/payload for the notification email, plus the recipient tag/subject.

## Deploying and Testing End to End
- Exporting the project: right-click project → Export → Next → Finish, producing a deployable JAR.
- Environmental Client ID/Secret only needs to be added to application properties if an API Manager policy (e.g., Client ID Enforcement) is applied — otherwise it isn't required. Client ID/Secret come from **Access Management**, tied to the app's domain name.
- To apply an API Manager policy to an app that's already deployed, you need to add the **Auto Discovery** element in Studio, then redeploy and re-export/re-upload the JAR — a Studio-side config change (e.g., linking a Global Element) requires re-deploying, since platform-side changes alone won't propagate into the already-running app.
- Deployment via Runtime Manager: Settings → Choose File → upload the exported JAR → Apply/re-upload → confirm "application started successfully" in the logs.
- End-to-end test: triggered a request via Postman with invalid target credentials configured on purpose → the request failed → flow routed to the global error-handling part → CloudHub Connector fired → a notification email was received, containing a hard-coded message ("issue with the target system") and, optionally, the original order payload for context.
- Also observed: a response coming back in Java/"grizzly" format traced back to a Transform Message not being converted to JSON — the same class of issue seen in the 14-03-2022 session, fixed the same way (convert via Transform Message before returning).
