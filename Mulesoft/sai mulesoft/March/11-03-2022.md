## Policies Recap: Auto Discovery, Client ID Enforcement, Exchange Credentials
- Auto Discovery links a Runtime Manager deployment to API Manager — deployment happens through Runtime Manager, while policy/security configuration happens in API Manager.
- The Client ID Enforcement policy is applied in API Manager, but the actual client ID/secret credentials are generated in Exchange, tied to a registered client application under a specific API ID.
- If a client ID/secret is lost after creating a client app, it can be retrieved again via Exchange → My Applications → select the application → view its credentials.

## Debug Logging Configuration
- To view debug-level logs (e.g. full payload output), go to the deployed application in Runtime Manager → Settings → Logging, and enable the relevant logging category there.

## Vertical vs Horizontal Scaling Recap
- Vertical scaling = vCore, used when a single request carries a large payload; default is 0.1 vCore, scalable up toward 32 GB.
- Horizontal scaling = worker count, used when the number of incoming requests is high; workers start from 0 (1 worker by default), and the platform's default worker limit is typically 8 (higher limits can be requested where needed).

## Error Handling (Flow-Level vs Global-Level)
- When a request fails inside a flow, execution routes into the flow's error-handling block.
- Two error handling components discussed:
  - **On Error Propagate**: stops processing and propagates the error upward/back to the caller.
  - **On Error Continue**: allows the flow to recover and continue processing after handling the error.
- Each error handler should have an explicit Error Type defined (e.g. an HTTP timeout error type) so that different failure causes — source system issues vs target system issues — can be distinguished and given a clear, meaningful message instead of a raw technical error (e.g. "could not resolve the address").
- On a genuine target-system failure, options discussed: return a clear, readable error message to the source system, and/or trigger an email notification about the failure.
- Error handling can be implemented at the flow level (custom logic per flow, needed when targets differ in format/conditions) or at the global level (shared, consistent handling across flows). The class built both to compare, with global-level handling generally preferred where flows behave similarly, for consistency.

## Multi-Target Practice Scenario (Order / Shipment / Trailer / Check-in)
- Practice task: build flows for multiple target systems, each with different data formats and trigger conditions.
  - **Order / Trailer**: JSON data; condition based on a source value that must equal 990 for the request to be forwarded — if the condition isn't met, the flow goes to a default block without raising an error; if the condition is met but the downstream call itself fails, that failure is handled with On Error Propagate.
  - **Check-in**: source data is XML, but the target expects JSON — requires an XML→JSON conversion step, followed by a further conversion into Java format for processing.
  - **Shipment**: condition based on an IB flag value that must equal 0 — data is only sent to the target system when this condition matches.
  - **Text-format target**: no condition attached — data is always forwarded to this target; since there's no branching logic to fall back on, any failure here is handled purely at the flow level.
- Best practices reinforced during the walkthrough: consistent naming conventions across resources/flows, proper logging at both debug (full payload) and info (a unique tracking field, e.g. outbound segment number or IB flag) levels, and securing the app via Client ID Enforcement.
