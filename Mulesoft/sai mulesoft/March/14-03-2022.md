## Conditional Routing & Variables
- Working flow had two routes/segments in the payload, e.g. `appointment_ping` and `appointment_ping_segment`. When reading a value from a specific route, you must reference the exact path for that route/segment in the payload — you can't just hardcode a generic field name.
- A **Set Variable** component was used to store the outbound segment number into a variable named `outbound_segment`. It is read dynamically from the incoming payload, not hardcoded.
- Reading variables in Mule 4 always uses the syntax `vars.variableName` (e.g., `vars.outboundSegment`).
- The same field/value can be used both as a stored variable (via Set Variable) **and** directly inside a condition check — you don't need to duplicate logic.
- Example condition: check if `payload.appointment.outboundSegment....warehouse` equals `990`; if true, the flow routes the request to the target system.
- Debugging approach: add a breakpoint, run the flow in Debug mode, use Postman to build/trigger the request in parallel while the app finishes deploying, then step through ("move it further") to inspect the Variables panel at each stage.
- If Studio shows "port not available" when starting a debug session, it usually means another instance of the app is already running — stop the existing run (red icon near the Mule debugger) before starting a new debug session.

## Flow-Level Exception Handling
- The demo intentionally pointed the target system at a deleted/unavailable deployment so the request would fail after the routing condition was satisfied.
- Error handling was implemented at the **flow level** using a **Try** scope around the risky HTTP call.
- The flow-level error handler declared three error-type blocks in this exact order: `HTTP:TIMEOUT`, another HTTP-related error type, and `ANY` as a catch-all.
- **Rule:** the `ANY` block must always be placed **last**. If `ANY` is placed first, every error (even ones that should hit a specific block) gets caught by it first, since blocks are evaluated top-down.
- In the test, the request timed out, so the flow landed in the `HTTP:TIMEOUT` block. Inside that block, a Transform Message component that was outputting Java-formatted data was fixed to output JSON before sending the error response back to the source system.
- A **Logger** component only prints to the console/log — it does not change or forward the payload. To actually send data back as a response to the source system, the payload itself must be transformed and returned (e.g., via Transform Message), not just logged.

## Logging & Log Retention
- Two logging levels were used: **INFO** (always printed) for lightweight, traceable fields like the outbound segment number, and **DEBUG** (only printed if the debug log category is explicitly enabled) for the full payload.
- Rule of thumb: never log the entire payload at INFO level — log only one key traceable field to avoid excessive memory/log usage.
- When troubleshooting a failed request, the first thing to ask the source system for is the **outbound segment number** — a unique ID generated per request that's used to trace the failed transaction in logs.
- Two ways to check logs:
  1. **Runtime Manager** application logs — limited storage capacity (mentioned as roughly 500 MB), shorter retention window.
  2. **Anypoint Monitoring** — logs retained for up to **90 days**, with no memory/storage limit mentioned for this option.

## Testing Tools & Transform Message
- Postman is functionally the same as what Mule documentation calls "REST Client" — officially **Advanced REST Client (ARC)**.
- When the target system returns an error in Java format instead of JSON, a **Transform Message** component is used to convert the response into JSON before it's returned to the caller.
- Returning a clear, properly formatted error message to the source system helps them quickly identify which transaction/request failed and report the correct details back.

  A: Yes — the same value can be referenced in a Set Variable and inside a condition expression at the same time.
