## Scatter-Gather Router
- Routers covered so far: **Choice** (condition-based routing), **Round Robin**, and now **Scatter-Gather**. (Note: Logger and Transform Message are "components," not routers; HTTP and other connectors are "connectors.")
- Choice router recap: used whenever there's a condition-based scenario — if the condition is true, the message goes to the target system; if not, the message is suppressed/ignored.
- **Scatter-Gather** is used when there is only **one source/one endpoint**, but **multiple target systems expect the same data**. It splits the single incoming request and sends it to multiple targets **in parallel** (simultaneously).
  - Example scenario used: a single API resource (e.g., "shipment" or "trailer") receives data from the source system, but two downstream targets — **Order** and **Shipment** — both need that same data.
- Scatter-Gather combines the responses of all target calls into a single combined output/response.
- It supports more than two targets — e.g., 3 or 4 targets can all be split to in parallel by adding more HTTP Request connectors inside the Scatter-Gather.

### Hands-on Demo Build
- Created a new Mule app named "scatter gather application."
- Flow: HTTP Listener (path `/scattergather`) → Logger ("data received from Postman," prints the payload) → Scatter-Gather component, containing two routes:
  - Route 1: HTTP Request connector → Order resource (method POST).
  - Route 2: HTTP Request connector → Shipment resource (method POST).
  - Logger added after each route to print the response payload from each target.
- It's a manual (non-RAML/non-API) application with a single endpoint: `http://host:port/scattergather`. It accepts both JSON and XML payloads.
- Testing in Postman: triggering the endpoint shows the request reaching the Mule app (visible in logger/console), then splitting in parallel — one branch to Order, one branch to Shipment.

### Media Type Mismatch Issue
- First test run: Order accepted the payload (JSON) successfully, but Shipment returned an **"unsupported media type"** error.
- Root cause: Shipment's target system only accepts **XML**, but the source system was sending **JSON** — Order accepts JSON so it worked, Shipment does not.
- Fix: Added a **Transform Message** component before the Shipment HTTP Request to convert the payload into XML before sending.
- After the fix, both targets returned success: "order placed successfully" and "shipment shipped successfully."

## Flow Variables (`vars`)
- A **variable** is used to store data temporarily so it can be accessed anywhere later in the same flow.
- Syntax to store: assign the incoming payload to a variable (e.g., named `data`) using a Set Variable / Transform component.
- Syntax to access: `vars.data` — this retrieves the full payload from anywhere in the flow.
- Scope: a flow variable is accessible across the **entire flow**, not just at one point.
- In the demo, the full source-system payload was stored into a variable called `data` right after it was received, so it could be reused later regardless of what transformations followed.

## Correlation ID
- Correlation ID is used purely for **tracking/debugging purposes** — not business logic.
- Every request that comes into a Mule application gets a unique **correlation ID** (also referred to as event ID).
- Real-world use case: if there are e.g. 1000 requests to an application in a day and a user reports an issue with one specific transaction, giving a **timestamp**, you can locate that request:
  1. In Runtime Manager, open the deployed application → **Logs**.
  2. Scroll to the reported timestamp and find the log entry.
  3. Copy the **event/correlation ID** shown there.
  4. Paste it into the log search bar and search — this filters logs down to only that one transaction's entries.
- Because the same correlation ID is attached to every log line generated by a single request (from JSON received → converted to XML → sent to target → target's response received), you can trace the full journey of just that one transaction, separate from the other 999 requests in the logs.
- Without correlation ID search, you cannot isolate a single transaction's logs — you'd have to scan everything.

## Round Robin Router (and Scatter-Gather comparison)
- **Round Robin** is another router used (like Scatter-Gather) for splitting a request to multiple targets — but it executes **sequentially**, not in parallel.
- Sequential execution: the first target (e.g., Order) is called first; only once that call succeeds does it move on to the next target (e.g., Shipment) — one at a time, in order.
- Scatter-Gather = parallel execution (all targets hit simultaneously, no dependency between them).
- Round Robin = sequential execution (one target after another, in sequence).
- Failure behavior difference (important for the upcoming error-handling topic):
  - In **Scatter-Gather**, since it's parallel with no relation between branches, if one branch fails the flow still proceeds/continues.
  - In **Round Robin**, since it's sequential, if there's an issue at one step, the flow does **not** proceed to the next step — it fails there.
- Choosing between them depends on the requirement: use Scatter-Gather when targets need data simultaneously; use Round Robin when targets need to be hit one after another in order.

## Error Handling (introduced, full topic upcoming)
- Example scenario discussed: what happens if the target system is completely down (application undeployed)?
  - The request from Mule will not reach the target at all.
  - You'd need to check the logs to determine which step executed last and why the final step didn't execute.
- Every programming language has an exception-handling pattern (e.g., Java); Mule has the same flexibility — errors occurring anywhere in the flow can be caught and handled, and a proper response/error can be communicated back to the source system.
