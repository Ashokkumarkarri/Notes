## Anypoint MQ: Manual Acknowledgement (ACK/NACK)
- When consuming a message from Anypoint MQ in **manual ack mode**, the flow receives an ACK token in the message attributes.
- If the message is successfully delivered to the target system, that ACK token is used to configure the **ACK** (acknowledge) at the end of the flow, removing the message from the queue.
- If delivery to the target system fails, the flow routes to error handling, where a **NACK** is configured instead — the message is not removed and can be picked up again.
- If the ACK is never configured (e.g., target session/connection issue), the message stays in the queue and will be picked up and retried again.
- Related: a Circuit Breaker approach was referenced as an option for handling repeated target-system failures.

## DataWeave: Removing Duplicates and Range Function
- Scenario: employee records where an "employee" value should always be unique within an organization; if duplicate employee records appear, they must be de-duplicated.
- A DataWeave function to remove duplicate values from an array/object was discussed for this purpose.
- **Range function**: used to generate a sequence of numbers (example given: 0 to 3) for use cases like generating employee IDs.

## MUnit: Global Error Handling
- Why write MUnit test cases when a flow can be tested with Postman: MUnit lets you test the flow internally, without connecting to the source or target system, by mocking connectors.
- To implement error handling globally, create a **new global configuration file** to act as a global error handler.
- In the MUnit test flow, mock the connectors — specifically the Listener and the Requester — and use **Set Event** to pass the expected payload, same approach as the main flow's happy-path test.
- In the mocked Requester component, there is an **Error** option that lets you raise a specific error type (e.g., set the error type to "ANY", or a more specific type like an HTTP timeout/connectivity error) to simulate a failure.
- By raising an error in the mock, you can drive execution into the global error-handling logic and verify it behaves correctly, without needing a real failure from the target system.
