## Object Store — Store and Retrieve Operations
- To persist data, use the **Store** operation of the Object Store connector: drag and drop it, and configure a **key** (acts like a variable name) to save the payload/value under.
- To read data back later, use the **Retrieve** operation, referencing the same key used when storing.
- Object Store connector configuration requires **Max Entries** and a max entry lifetime:
  - Default retention is **30 days** — after that, stored data is automatically deleted.
  - This is customizable based on business requirement (e.g. 1 day, 2 days, 10 days, etc.).
- A flow needs a trigger to run store/retrieve logic — options used: **HTTP Listener** or **Scheduler**.
- Demo: a POST call stored `car` and `price` values in the Object Store; a separate flow with a GET call (no request body) retrieved the same data using the Retrieve operation.
- Because data is stored in the Object Store rather than passed directly between flows, any number of applications/flows can consume it later via Retrieve + the correct key — this decouples the storing flow from the consuming flow (they don't need a direct Flow Reference or synchronous call between them).

## Using Object Store to Cache Static/Constant Data
- Example scenario: an integration receives per-request dynamic data (e.g. `car`, `price`) from a source system, and also needs a value from an external system (e.g. Salesforce) that rarely changes — e.g. `country = India`. This second value is "static data."
- Problem with querying the external system (Salesforce) on every single request:
  - It's wasteful, since the value never changes.
  - If the external system (Salesforce) goes down, the whole flow/thread fails, even though the needed value hasn't changed.
- Solution pattern demonstrated:
  - A separate scheduled flow queries Salesforce infrequently (demo used a **Fixed Frequency** schedule of **30 days**, i.e. roughly once a month) and stores the result in the Object Store under a key (e.g. `data`).
  - The main flow, for every incoming request, does **not** call Salesforce directly — it retrieves the cached static value from the Object Store, combines it with the dynamic payload from the source system using **Transform Message**, and sends the combined payload to the target system.
  - Net effect: the external system is called only once per scheduling interval (e.g. once a month) instead of on every transaction.
- Continued in the second recording of the day: deployed the same setup to a sandbox environment and re-verified the pattern — dynamic values (`car`, `price`) come from the source system per request, while the static value (`country`) is served from the Object Store cache instead of being re-fetched from the external app each time.
- Core takeaway: **Object Store is primarily used to store/cache static (constant) data** so that it doesn't need to be re-fetched from an external system on every transaction.

## Doubts / Q&A
- **Q: When should you use static data storage (Object Store) in a flow?**
  **A:** For message reliability — so the flow doesn't depend on an external system being available and responsive for every single transaction.
