## HTTP vs HTTPS Configuration (Mule 3 vs Mule 4)
- In **Mule 3**, switching a listener's protocol from HTTP to HTTPS requires configuring certificates — there are always two: **Truststore** and **Keystore**, provided along with a password, configured inside `src/main/resources`.
- In **Mule 4**, this is not mandatory — you can directly change the protocol from HTTP to HTTPS on the listener without configuring certificates separately.
- If the protocol is changed to HTTPS, the client must call the HTTPS URL — triggering the old HTTP URL will not work and the request won't go through.

## API-Based vs File-Based Integration
- API-based systems support only the **HTTP** and **HTTPS** protocols; the flow starts with an **HTTP connector/listener**.
- API-based flows are tested using **Postman** (configuring URL, method, and host).
- File-based integration example requirement: a source system (e.g. Salesforce) has a file at some location — the file needs to be read and sent to a target system.
- To transfer/read files, Mule Palette provides connectors: **File**, **FTP**, **SFTP** (and **FTPS**).
  - **FTP** = File Transfer Protocol.
  - **SFTP** = Secure Shell File Transfer Protocol — fully secure; ~99% of organizations use SFTP for file-transfer requirements.
  - A secure FTP (FTPS) connector is not available by default in the palette — it needs to be added from **Anypoint Exchange**.
- Postman **cannot** be used to test FTP/SFTP/FTPS-based flows since they are not HTTP-based. Instead, test by placing a sample file locally and letting the flow pick it up.

## File Connector Demo & Scheduling Strategy
- Built a demo "file transfer" project: dragged the **File connector**, configured it, and set the **Directory** field (the path where the file is located), e.g. a `file` folder on the Desktop containing a `sample` file.
- **Scheduling Strategy** on the File connector — two options: **Cron Expression** or **Fixed Frequency**.
- Fixed frequency example: `1000` milliseconds = 1 second. With this set, the connector checks the folder every second for a new file.
- In practice, the frequency (every second/minute/hour, etc.) is set according to the business requirement — if a file is found, it is read.
- After reading, the file's content can be sent to a target system (configure target credentials); conditional logic on the payload can be handled with a **Choice** router.
- Tested with a breakpoint + debug run, printing data via a **Logger**; sample file content in the demo had 5 lines.
- **Auto Delete** option on the File connector:
  - Default is `false` — the file remains in the folder after being read, which causes it to be picked up again and again (duplicate processing). E.g. 10 files left undeleted would keep getting re-iterated.
  - Setting it to `true` automatically deletes the file once it has been picked up and delivered to the target system, preventing duplicate reads.

## Scheduling: Fixed Frequency vs Cron Expression
- Two scheduling strategies: **Fixed Frequency** and **Cron Expression**.
- Cron expressions can be generated using an online tool called **CronMaker**.
- Demo: scheduled a flow to run weekly, every Monday at 11:00 AM — selected day (Monday) and time (11:00) in CronMaker, then clicked **Generate** to get the expression; the generated expression was copied into the connector config.
- Timezone matters when setting a cron expression (e.g. IST vs UTC) — set according to business requirement.
- With a weekly Monday 11 AM cron schedule, the flow runs once per week, at that exact time, until the next Monday.
- Scheduling also applies to API-based systems — an HTTP Requester call can be scheduled using GET (retrieve data from an external system) or POST (send data to a system).
- Connector naming across versions:
  - **Mule 4**: called the **Scheduler** connector.
  - **Mule 3**: the equivalent was called **Poll** (a "poll job").
  - Only the naming changed between versions — the underlying options (Cron Expression and Fixed Frequency) remain the same.
- The Scheduler was noted as the fourth connector covered in the course so far, after HTTP, Anypoint MQ, and a (Cloud) Notification connector.

## Encrypting Sensitive Data in Property Files
- In **Mule 3**, sensitive values in a property file are encrypted using an encryption algorithm — example given: **AES**.
- If a property value is encrypted but the corresponding secure-properties configuration is not set up in **Global Elements**, the application will fail to deploy.
- The Global Elements configuration needed:
  - **Mule 4**: called **Secure Property Config** (previously referred to loosely as "secure property placeholder config").
  - **Mule 3**: called **Secure Property Placeholder**.
  - Only the naming changed between Mule 3 and Mule 4 — the concept (encrypt property values, then reference them via a secure-properties config in Global Elements) is the same.
