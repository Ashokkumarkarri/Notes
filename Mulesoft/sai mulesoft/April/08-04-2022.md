## DataWeave: String Concatenation Operator (++)
- Sample scenario: input payload is a JSON array received from a source system, containing `firstName` and `lastName` fields.
- Requirement: target output should contain a single `name` field combining first name and last name.
- In Transform Message, set the key as `name` and build the value using the concatenation operator `++`.
- Syntax: `$.firstName ++ " " ++ $.lastName` — a space (empty string literal) is concatenated between first and last name.
- The `++` operator works across types: when concatenating a number and a string, DataWeave automatically converts and adds them together as output.

## DataWeave: Split By Operator
- Second scenario: source system sends a single `name` field whose value actually holds a combined string (e.g., a location value), but the target only needs the first word/segment.
- To extract just the first word, use the `splitBy` operator, splitting on the space character (double quotation with a space inside).
- After splitting, take the first element of the resulting array as the target value.

## Debugging: Address Already in Use
- Error encountered while running/debugging a flow: "Address already in use."
- Cause: multiple Mule apps running on the same port (port conflict).
- Fix: stop the currently running apps and restart Anypoint Studio, then re-run the flow in debug mode.

## MUnit Testing: Mocking Connectors & Set Event
- Flow structure discussed: HTTP Listener → Logger → Flow Reference (calls a main flow) → Requester (connects to target system).
- In the test flow, the Requester connector was mocked so it does not actually connect to the target system.
- Logger component is used purely to print/inspect the payload for verification.
- Use the **Set Event** component to pass an expected payload into the flow, simulating what the flow would normally receive.
- This lets you validate the flow's logic end-to-end without depending on real connectivity to the target system.
- Discussion: Postman can also test a flow, but a Postman request actually reaches the real target system. MUnit, by contrast, mocks connectors and uses Set Event to test the flow purely internally — no live connection to source or target needed.
- A test target system was intentionally left invalid/unreachable; because the Requester was mocked, the test still passed instead of failing, since it never truly attempted the connection.

## MUnit Code Coverage
- After running MUnit tests, a coverage report can be generated.
- Coverage is calculated from how many flow components were exercised by the test — e.g., a flow with Listener, Logger, Transceiver (Requester) components covered by tests shows 100% coverage when every component is hit.
- Coverage is important in real organizations: deployments run through an automated CI/CD pipeline (not manual jar upload to Runtime Manager, which is only used casually).
- The pipeline validates MUnit code coverage before allowing deployment — commonly required to be above 80% (standards vary by organization: some require 70%, some 80%, some 85%).
- If coverage is below the required threshold, the pipeline rejects the deployment and the application will not go live.
