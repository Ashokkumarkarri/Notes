## File/FTP Streaming Strategies
- Base scenario: reading a file from an SFTP location and sending it to a target system. Since there's no transformation needed, a **Request** component is used (not Transform Message). Loggers are added after reading the file and again after sending it to the target system, for tracking.
- FTP Connector has an **Advanced** section exposing **Streaming Strategies**, with two types: **Repeatable** and **Non-repeatable**.
- Default behavior (no streaming strategy configured): Mule tries to read the entire file into memory at once. For a large file (example used: 10 GB), this throws an **Out Of Memory** exception because available memory is much smaller than the file size.
- **Repeatable streaming**: reads the file in chunks based on a configured buffer size (example: 512 MB), connecting to the file location repeatedly to pull one buffer-sized chunk at a time until the whole file is read in pieces.
- There is a **maximum buffer size** setting (example discussed: cannot exceed roughly 1 GB / 1024) — similar in concept to a max limit/TTL setting for the buffer.
- With streaming, Mule can process files of virtually any size (example: 50 GB) by reading and forwarding them in chunks rather than all at once.
- Example walkthrough: a 10 GB file read with a 1 GB buffer connects to the file location 10 times, sending 1 GB per attempt to the target system.

## Handling Large Payloads: Batch Job vs For Each
- Problem: even after streaming reads a chunk (e.g., 1 GB) from the source, the **target system's single-request capacity** may be much smaller (example: only 100 MB per request). This mismatch means the already-read chunk must be split further before sending.
- Two components are available in the Mule palette to solve this: **Batch Job** and **For Each**.
- **For Each**: processes records sequentially, one at a time (or in configurable batch sizes), pushing each to the target system before moving to the next.
  - Default block/batch size in For Each = 1 record per iteration.
  - Demonstrated with a 5-record file: with batch size 1, the loop executes 5 times, one record pushed per iteration (fully sequential).
  - With 100 records and a batch size of 5, the loop executes 20 times (100 ÷ 5).
- **Batch Job**: unlike For Each, Batch Job executes in **parallel** rather than sequentially.
  - Default batch block size in Batch Job = 1 (customizable, e.g., set to 100 records per block).
  - Recommended when dealing with very large record counts (example: more than 1 crore / 10+ million records, or ~100 GB of data), where a sequential For Each loop would require an extremely high number of iterations and introduce significant delay before reaching the target system.
  - Rule of thumb: for huge payloads where sequential processing (For Each) is too slow, use **Batch Job** for parallel execution instead.
- General guidance: rather than increasing compute (vCore) to handle bigger payloads, prefer streaming plus Batch Job/For Each to chunk and control how much data is sent per request to the target system.

## Doubts / Q&A
- **Q: Can we just configure Repeatable streaming directly with a 100 MB buffer instead of using Batch Job/For Each?**
  A: Yes, that is also a valid option for moderate payload sizes. However, for very large payloads (example given: 100 GB), a small buffer size means the connector reconnects to the file location a very large number of times (e.g., ~100 times), which increases the risk of connectivity issues — if a connection fails partway, the entire execution fails. Recommendation: test with a smaller payload (e.g., 100 MB) first to confirm the approach works reliably before scaling up; if repeated connectivity issues occur at scale, use Batch Job or For Each instead.
