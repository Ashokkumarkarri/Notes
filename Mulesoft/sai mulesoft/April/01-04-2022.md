## Mule 3 vs Mule 4 Message Structure
- Mule 3 uses inbound/outbound properties; Mule 4 replaces these with a single **Attributes** object.
- Mule 3 has three types of variables; Mule 4 simplifies this to a single **Flow Variable**.
- Payload works the same way in both versions.
- Mule 4 adds something not present in Mule 3: **Correlation ID**.

## Circuit Breaker Logic & Error Handling
- Circuit breaker logic is used to handle situations where the target system is down, so the flow doesn't keep failing uselessly against a dead target.
- Explained via a scenario: the target system is down, and there is a message waiting in the queue.
- Acknowledgement modes when consuming from a queue:
  - **Manual mode** — a message is only removed from the queue when it is explicitly acknowledged. The acknowledgement token is configured at the end of the flow, inside the exception-handling section.
  - **ACK** — message processed successfully → deleted from the queue.
  - **NACK** — message processing failed → message is **not** deleted from the queue, so it is picked up and retried again.
- Walkthrough: target system is down → message is picked from the queue → delivery attempt to the target fails → error handling routes to NACK → message stays in the queue → the same message is picked again on the next cycle, repeating until it succeeds (or is otherwise handled).
- Circuit breaker settings are configured under the connector's **Advanced** section ("Edit inline"):
  - **Error Types** — which target-system errors should be treated as expected failures (examples given: timeout, connectivity, unauthorized).
  - **Error Threshold** — the peak number of consecutive failures allowed before the circuit trips (example used: threshold = 5, i.e. 5 consecutive timeout errors trips the flow).
  - **Trip Timeout** — how long the circuit stays tripped before allowing requests through again; configured as a duration (the class floated both milliseconds and minutes as possible units — exact value depends on configuration).
- MQ publish/subscribe was demonstrated using an existing demo project ("mq"): one flow publishes a message, another flow subscribes to and consumes it.

## Anypoint MQ — Acknowledgement Modes & Dead Letter Queue (DLQ)
- Messages that cannot be processed correctly ("bad data") can be routed to the **DLQ (Dead Letter Queue)** instead of being retried indefinitely.
- **DLQ = Dead Letter Queue** — holds messages considered bad/undeliverable; these can be inspected and removed separately from the main queue.
- Queue status (e.g. message count, "zero messages") can be checked directly on the queue.
