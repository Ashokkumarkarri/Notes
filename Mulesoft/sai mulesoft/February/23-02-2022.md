## Recap: API-Level Restrictions and Status Codes
- Restrictions can be configured directly inside an API, for example:
  - Accept only JSON data, or only XML data.
  - Accept only a specific HTTP method (e.g., only POST, or only GET).
- Status codes tied to these restrictions:
  - Wrong/unsupported HTTP method → **405 Method Not Allowed**.
  - Wrong resource/endpoint triggered → **404 Not Found**.
  - Wrong media type triggered (e.g., sending XML when JSON is expected) → **415 Unsupported Media Type**.
  - Successful request → **200 OK**.

## REST vs SOAP (Recap)
- **REST** = Representational State Transfer.
  - Built using **RAML** language.
  - Supports all media types (XML, JSON, CSV, Java, text, etc.).
  - Supports all HTTP methods.
- **SOAP** = Simple Object Access Protocol.
  - Supports only XML media type.
  - Supports only the POST method.
- REST is generally preferred for its flexibility compared to SOAP.
- RAML = **RESTful API Modeling Language**.
- REST APIs are developed inside **Design Center**, which sits inside **Anypoint Platform**.

## Creating a New API in Design Center
- In Design Center, click **New API** and give it a name.
- Naming convention: the API name should reflect the source system it's integrating with.
  - Example: source system is Salesforce → name the API "Salesforce API".
  - Example: source system is a database → "Database API".
  - Example: source system is a Java-based system → "Java space API".
  - Example used in the live demo: source system "Amazon" → API named "Amazon API".
- In real-time projects, naming based on the source system is treated as a rule/convention, not optional.

## RAML Structure
- When a new API is created, RAML **version** and **title** are auto-populated.
  - Title is generated automatically based on the API name (e.g., "Amazon API" → title "Amazon API").
  - RAML version history discussed: an older version (0.8) was used a few years back; the version referenced for the newly created API in this session was more recent (discussed as 1.0, later also referred to as 1.4 in the same discussion).
- **Version** (e.g., `v1`, `v2`, `v3`) is defined based on convenience, but standard practice is to start every API at version 1 (`v1`).
- **Base URI**: includes protocol and media type configuration.
  - **Protocols**: define which protocol(s) the API accepts from the source system (e.g., HTTP, HTTPS).
  - **Media types**: define which data formats the API accepts, using MIME-type syntax:
    - JSON → `application/json`
    - XML → `application/xml`
    - Text → `text/plain`
    - Java → `application/java`

## Defining Resources, Methods, and Status Codes in RAML
- Below the base configuration, the RAML defines **resources** — these represent the API's endpoints/paths (e.g., `/order`, `/demo`, `/dummy`).
- If a source system triggers a resource/path that doesn't exist in the RAML → **404 Not Found**.
- Each resource defines which HTTP **method** it accepts (e.g., only POST).
  - If the wrong method is used → **405 Method Not Allowed**.
- Each resource also defines the **body**/media type it accepts (e.g., only JSON).
  - If the wrong media type is sent → **415 Unsupported Media Type**.
- A correctly formed request (right resource, right method, right media type) returns **200 OK**.
- Extra blank lines/spacing in the RAML schema are not mandatory — they're only used for readability.

## Importing RAML into Anypoint Studio and Generating API Kit Router Flows
- Once a RAML is designed in Design Center, it can be imported into Anypoint Studio when creating a new Mule project (option available during project creation to import/browse and publish the RAML).
- Demo project created: "Amazon Mule Integration".
- After importing the RAML, Anypoint Studio **automatically generates the flows** matching the RAML's resources/methods — no need to manually drag and drop components from the palette.
- These auto-generated flows are called **API Kit Router flows**, because they are generated from the REST API definition and routed via the API Kit Router.

## Testing the API via Postman
- HTTP Listener configuration for the demo: host `0.0.0.0`, port `8081`.
- Listener path pattern: `/api/*` — the `*` represents the resource name (e.g., `/api/order`).
- Test scenarios run against the `order` resource (configured to accept POST + JSON only):
  - **GET** request instead of POST → **405 Method Not Allowed** (method restriction enforced by the API).
  - **POST** with **XML** body (API expects JSON) → **415 Unsupported Media Type**.
  - Wrong/incorrect resource path → resource not found error.
  - **POST** with correct **JSON** body → **200 OK**, success response returned.
- Postman is used purely as a testing tool standing in for the real source system (e.g., "Amazon") — there is no real source system integration in the demo.

## Adding a Second Resource and Deploying to CloudHub
- A second resource, **shipment**, was added to the same API (copied from the `order` resource setup).
- Each resource in the RAML generates its own dedicated flow — e.g., "order flow" and "shipment flow".
- To move from local testing to CloudHub testing, the `localhost` and local port in the request URL are replaced with the CloudHub deployment URL/host.
- After deployment, the app was tested again via Postman against CloudHub:
  - Wrong media type to `shipment` → error (unsupported media type).
  - Correct JSON to `shipment` → **200 OK**, success message.

## Why APIs Are Needed: Restrictions vs Security
- A Mule application can technically be built without an API layer (just Listener + Logger + other components), but doing so gives **no restrictions and no security**.
- The API layer is what allows adding restrictions: method-level restrictions, media-type-level restrictions, and business-level restrictions.
- APIs alone (as configured so far) provide **restrictions**, not full **security** — actual security is applied separately.
- **Security** is configured in **API Manager** (inside Anypoint Platform).
  - API Manager lets you see and manage each API's URL, allowed methods, and media types.
  - Under API Manager there's a **Policies** section with options such as: client ID enforcement, basic authentication, rate limiting, throttling.
  - Detailed policy configuration was deferred to upcoming sessions.

## Debugging Mule Applications (Debug Mode)
- Running an application normally only shows the final result (data reaches the app, flow executes, response returned) — it doesn't show what happens at each component.
- **Debug mode** allows checking the flow at a **component-to-component level** — how data moves from the Listener → API Kit Router → the specific resource flow.
- To use debug mode:
  - Add a **breakpoint** on the relevant component/flow in Anypoint Studio.
  - Right-click the project → **Run in Debug Mode**.
  - When a request is sent, execution pauses at the breakpoint.
  - The **payload** (the actual data sent by the "source system") can be inspected while paused.
  - A "resume" control (referred to as the arrow/icon in the top-right corner) is used to step the execution forward to the next component.
- Debug mode demo:
  - Triggered data for the **shipment** resource → request paused at the breakpoint on shipment flow; payload visible.
  - Triggered data for the **order** resource with XML (order resource expects JSON) → request reached the API Kit Router, but failed once moved forward, because the API definition enforces JSON-only for that resource.
  - Triggered data for the **order** resource with correct JSON → request routed through the API Kit Router into the **order flow**, completed successfully, and a 200 response was visible in Postman.
- Routing logic: the **API Kit Router** is responsible for directing an incoming request to the correct flow (order flow vs shipment flow) based on the resource name in the request path.
- Debug mode runs the exact same logic as a normal run — the only difference is that execution can be paused and inspected step by step.
