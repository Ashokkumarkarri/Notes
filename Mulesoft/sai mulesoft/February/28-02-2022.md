## Mule 4 vs Mule 3 Message Structure
- **Mule 4 message structure** components:
  - `attributes` — metadata (headers etc.), replaces Mule 3's inbound/outbound properties
  - `correlationId` — unique ID
  - `payload` — same concept in both versions
  - variables — Mule 4 has only **one** type of variable, the **flow variable**
- **Mule 3 message structure** components:
  - `inboundProperties` / `outboundProperties` — replaced by `attributes` in Mule 4
  - `payload` — unchanged
  - variables — Mule 3 has **three** types: **flow variable**, **session variable**, **record variable**
- Mule 3 does not have the concept of a Correlation ID (that concept is specific to Mule 4).
- Correlation ID: a unique ID automatically generated by Mule itself in real time for every incoming request.

## Accessing Payload, Headers, and Variables
- Sample app built to demonstrate: one **HTTP Listener** + one **Logger**, configured with host/port/path, run in **Debug** mode, tested by sending a request with both body and header data from Postman.
- **Payload** — data coming in the request body:
  - Mule 4 syntax: `#[payload]`
  - Mule 3 syntax: same, `#[payload]` (no change)
- **Headers** — data coming in the request header, accessed via attributes:
  - Mule 4 syntax: `attributes.headers.<key>` (demoed with a header key, e.g. `attributes.headers.bat`)
  - Mule 3 syntax: `inboundProperties.headers.<key>` (e.g. `inboundProperties.headers.bat`)
- **Variables**:
  - Mule 4 (flow variable only): `vars.<variableName>`
  - Mule 3 (flow variable): `flowVars.<variableName>` — note the capital **V** in `flowVars`
  - Mule 3 also has session and record variables (flow, session, record — three types total), though only flow variable syntax was demoed.
- Without a **Logger**, nothing can be tracked/printed in the console — the app will still return a `200` response, but you won't be able to see what data passed through.

## Set Variable Component
- Demonstrated dragging a **Set Variable** component into the flow to assign a value to a variable.
- Example shown: a variable named `country` assigned the value `"India"`.
- The assigned variable can then be accessed and printed later in the flow (e.g., in a Logger) using `vars.country`.

## Task: Print Credentials and Payload via Logger
- Using the existing application (already secured with Basic Authentication / Client ID Enforcement from earlier sessions), the task was:
  1. In the **first Logger**, print the **username and password** (or client ID/secret, depending on which policy is applied) — these arrive in the request header and must be accessed via `attributes`.
  2. Add a **second Logger** to print the **actual payload** coming in the request body from the source system (in whatever format it arrives — XML or JSON).
- Point emphasized: whatever policy is applied to the API (Basic Auth or Client ID Enforcement), the corresponding credentials should be printed using this logger pattern.

## Manual Application vs API-Based Application Design
- Mule apps can currently be built over **HTTP** (an HTTP Listener is often called the "HTTP connector"). Other available protocols mentioned: SFTP, FTPS, FTP, MFT.
- Two ways to design an HTTP-based Mule app:
  1. **Manual application** — manually drag-and-drop a Listener and Logger (and other components) from the palette.
     - No restrictions can be enforced: any HTTP method (GET/POST/DELETE/etc.) and any media type will be accepted.
  2. **Via API (RAML-based)** — flows are auto-generated from a RAML definition; these are called **APIKit Router flows**.
     - REST APIs normally support all media types and all HTTP methods, but with RAML you can explicitly restrict which methods/media types a given resource accepts (e.g., only POST, only JSON).
     - Restrictions can only be enforced through the API-based (RAML/APIKit) approach — not through a manually built flow.
- End-to-end flow to secure an API (recap):
  1. Design the RAML in **Design Center**.
  2. Publish the RAML to **Exchange**.
  3. Import it into **API Manager**, where an API instance is auto-generated.
  4. Apply a **policy** in API Manager based on the organization's standard (Basic Authentication or Client ID Enforcement).
  5. Configure **Auto Discovery** (with the API ID) in the Mule app so it links to this API instance.
  6. Deploy the application to **Runtime Manager**.
  7. Configure the environment-level **Client ID / Client Secret** in the app's Runtime Manager settings — only then does the API move to **Active** status in API Manager and the policy actually gets enforced.
  8. Note: policies configured in RAML/Design Center are for documentation/design purposes only — editing the API in Design Center does **not** automatically affect the already-deployed application.

## Doubts / Q&A
- **Q: What is the difference in message structure between Mule 4 and Mule 3?**
  A: Mule 4 has `attributes`, `correlationId`, `payload`, and a single `vars` (flow variable). Mule 3 has `inboundProperties`/`outboundProperties`, `payload`, and three variable types — flow (`flowVars`), session, and record variables.
- **Q: Where can restrictions (like accepting only JSON or only POST) be applied — manual flow or API flow?**
  A: Only through the API (RAML/APIKit) based flow. A manually built flow (Listener + Logger dragged from the palette) accepts any method and any media type with no restrictions possible.
