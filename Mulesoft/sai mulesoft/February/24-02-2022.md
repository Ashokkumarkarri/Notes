# MuleSoft Training — 24-02-2022

## Topics Covered
- API Auto Discovery (linking API Manager and Runtime Manager)
- Basic Authentication policy
- Configuring Client ID / Client Secret in application settings
- Deploying an application to CloudHub via Runtime Manager
- Testing a secured API with Postman

## API Auto Discovery
- Auto Discovery is a component (searchable in Anypoint Studio's palette) used to link an API designed/managed in **API Manager** with the actual running app in **Runtime Manager**.
- Steps to configure:
  - Copy the **API ID** from API Manager.
  - In Anypoint Studio, go to Global Elements → create a new **Auto Discovery** component.
  - Specify the API ID in the Auto Discovery config and select the main flow.
- Once Auto Discovery is configured, policies applied in API Manager (e.g. Basic Authentication) get enforced on the deployed runtime application.
- Important rule: **policies only work when the app is deployed via Runtime Manager** (CloudHub) — they do **not** apply when running/testing the app locally in Anypoint Studio.
- If you want to test the application locally, temporarily remove the Auto Discovery component; once the flow logic works as expected, add Auto Discovery back and redeploy so the policy takes effect.

## Basic Authentication Policy
- Applied in API Manager on top of the API that was designed in Design Center.
- Secures the application with a username and password.
- Without the correct credentials, requests return `401 Unauthorized` (client "not agreeing with the password" / invalid credentials).
- To call a Basic-Auth-secured API from Postman: go to the Authorization tab, set type to **Basic Auth**, and enter the username and password.

## Configuring Client ID and Client Secret
- Even when using Basic Authentication, an additional environment-level credential pair — **Client ID** and **Client Secret** — must be configured for the application to work with API Manager policies.
- Location: open the deployed application in Runtime Manager → **Settings** → **Properties** → configure Client ID and Client Secret there.
- Until this is configured, the API stays in **"Unregistered"** state in API Manager; once configured correctly, it moves to **"Active"** state and the policy gets applied to the application.
- A small spelling mistake in these property keys will prevent the application from deploying correctly.

## Deploying to Runtime Manager (CloudHub)
- Steps demonstrated:
  1. In Anypoint Studio, right-click the project → **Export** → confirm the export location (project saved locally, e.g. C:\Users\...) → **Finish**.
  2. Go to the Anypoint Platform → **Runtime Manager** → open the **Sandbox** environment.
  3. Click **Deploy Application**, give it a name (example used: "amazon integration"), upload the exported file, and deploy.
  4. Application goes through a **Deploying** state before reaching **Started**.
- Check environment name (e.g. "Sandbox") when configuring environment-specific Client ID/Secret.

## Testing the Secured API
- Once the app shows **Started** in Runtime Manager and the API shows **Active** in API Manager, the policy (Basic Auth) is linked to the running application.
- Test flow:
  - Copy the CloudHub application endpoint from Runtime Manager.
  - In Postman, replace localhost/local port with this cloud endpoint.
  - Sending a request **without** username/password returns `401 Unauthorized` — confirms Basic Authentication is active.
  - Adding the correct username and password in Postman's Basic Auth tab allows the request through.

## Doubts / Q&A
- **Q: What policy did we apply to secure the API?**
  A: Basic Authentication — secures the Mule app using a username and password.
- **Q: If a policy is not applied, what happens?**
  A: The application can still be deployed and tested directly without any additional client ID/secret configuration — that step is only required once a policy (like Basic Auth or Client ID Enforcement) is applied.
