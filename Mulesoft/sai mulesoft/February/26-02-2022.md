## Managing Deployed Applications
- In Runtime Manager, a deployed application can be **started**, **stopped**, or **deleted** using the dropdown options on the application.
- A **trial account** on Anypoint Platform allows a maximum of **10 deployed applications**.
- Unwanted/unused applications should be deleted (via the delete option) to stay within this limit.
- To trigger a request against a Basic-Auth-secured application, the correct username and password must be supplied — without them the request will not go through (this was a recap check during the session).

## Client ID Enforcement Policy
- New task given: create an API with one resource that accepts text data (following the same process as the previous Basic Auth exercise).
- Task steps:
  1. First secure the app with **Basic Authentication** policy and get it working (deploy and test successfully).
  2. Then **remove** the Basic Authentication policy.
  3. Apply the **Client ID Enforcement** policy instead, in API Manager.
- Key difference from Basic Authentication:
  - Basic Authentication: you manually enter a **username and password** directly inside the policy configuration in API Manager.
  - Client ID Enforcement: the app is secured using a **Client ID and Client Secret**, which are **not** entered manually in the policy — they must be **generated in Anypoint Exchange**.
- To generate Client ID / Client Secret:
  - Open the API in **Exchange**.
  - Use the **three-dot menu** (top-right corner) → look for **"Request Access"**.
  - This generates the client credentials tied to that API in Exchange.
- Where each thing is configured:
  - **Policy** (Basic Auth or Client ID Enforcement) → applied in **API Manager**.
  - **Client ID / Client Secret** → generated in **Anypoint Exchange**, not typed manually.

## Intro to Real-Time Integration Concept
- General integration pattern discussed: a source system sends data, and some other system(s) act as the target.
- A Mule application sits in between and establishes connectivity between the source and target systems — this is real-time integration.
- This was introduced as a lead-in to upcoming, more detailed real-time integration practice.

## Doubts / Q&A
- **Q: Basically, what is the difference — what do we do with the basic policy?**
  A: Remove the Basic Authentication policy and apply the Client ID Enforcement policy instead.
- **Q: Where do we apply the policy?**
  A: In API Manager.
- **Q: What are the credentials involved?**
  A: For Basic Authentication it's username and password (entered directly in the policy). For Client ID Enforcement it's Client ID and Client Secret, which must be generated in Exchange rather than entered in the policy screen.
