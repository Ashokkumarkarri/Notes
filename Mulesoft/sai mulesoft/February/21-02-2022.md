## HTTP URL Structure
- Basic URL formation (Universal Resource Locator): `protocol://hostname:port/path`
- Example breakdown: protocol, then `://`, then hostname, `:`, port, `/`, path.
- An HTTP endpoint will always follow this pattern: `protocol://host/path`.
- Understanding this formation is essential for triggering data correctly to a Mule application using Postman.

## Testing a Mule Application with Postman
- Postman is used as a stand-in "source system" for testing — in real time, this role would be a live application, but for practice, Postman sends the test data.
- Steps: open the HTTP Listener in Anypoint Studio to check its configured URL (protocol, host, port), open Postman, paste the URL, and trigger a request.
- After deploying/running the app, check the console in Anypoint Studio to confirm data was received.
- A response of **200** in Postman indicates the request was successful and the data reached the Mule application.
- Multiple trigger attempts can be made from Postman to confirm consistent behavior.

## Deploying a Mule Application to CloudHub (Runtime Manager)
- Anypoint Platform (including Runtime Manager) runs in the cloud, hosted on AWS.
- Two ways to deploy from Anypoint Studio:
  1. Right-click the project → **Deploy to Cloud Hub** (links directly to your Anypoint Platform account).
  2. Right-click the project → **Export** → Next → Finish. This saves the project as a `.jar` file (e.g., to the C: drive, such as `dummy-app.jar`).
- To deploy an exported `.jar` manually: log in to Runtime Manager → Deploy Application → give the application a name → Choose File → Upload File → select the `.jar` from its saved location → Deploy Application.
- After deployment, the application shows a "deploying" state, then moves to "started" once live.
- Runtime Manager has multiple environments, including **Design** and **Sandbox**.
- A trial account can be used to get started; an additional account can be created later using a different email if needed.

## Local Testing vs Runtime (CloudHub) Testing
- Example listener configurations in Anypoint Studio use `localhost` and a local port by default — these only work for **local testing**.
- To send data to the application once it's deployed and running in CloudHub, `localhost` and the local port must be **replaced with the CloudHub URL**.
- The CloudHub URL for a deployed application can be found under the application's Settings in Runtime Manager — copy this URL and use it in Postman in place of `localhost:port`.
- Care should be taken with the exact URL formatting (e.g., no extra spaces) when constructing the request.

## HTTP Status Codes
Noted as fixed, standard codes — they apply regardless of which technology (not specific to Mule):
- **200** — Success
- **400** — Bad Request
- **401** — Unauthorized
- **403** — Forbidden
- **404** — Resource Not Found
- **405** — Method Not Allowed
- Unsupported Media Type (code number not stated in session)
- **500** — Internal Server Error
- **502** — Proxy Error (Bad Gateway)
- **503** — Service Unavailable
- **504** — Timeout

## Hands-on Task: Salesforce Integration Application
Assigned task, to be completed alongside finishing the software installation:
1. Open Anypoint Studio → File → New → create a new **Mule Project**.
2. Name the application **Salesforce Integration** (or a more readable variant like "Salesforce mule integration").
3. Set the project path using your own name (e.g., `/venkat`, `/mahesh`, `/lokesh`) rather than leaving the default.
4. Drag and drop a **Logger** component onto the flow.
5. In the Logger, set the message to: `Message received from source system` followed by `#[payload]` (to log the actual payload).
6. Run the application locally first, and test it using Postman (local testing).
7. Deploy the application to CloudHub via Runtime Manager.
8. Test again using Postman, this time using the CloudHub URL instead of localhost, to confirm the runtime deployment works end to end.
